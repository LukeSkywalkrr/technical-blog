S01 Start from the customer

What actually happens when you pay with a card?

You see one form and one button. Behind that button is a carefully split system designed so that most of the merchant's application never sees the card number.

Imagine you are buying headphones from shop.example.com. You type your card number, expiry date, and CVC, then press Pay ₹4,999. The page looks like one website, but the payment fields may not belong to the merchant at all.

1. Type cardinside a PSP hosted field
2. Send securelydirectly to the PSP
3. Create referencetoken / payment method ID
4. Merchant paysamount + reference
5. PSP resolvesinside its card data boundary
6. Bank authorizesapprove / decline
7. Status returnsmerchant sees result, not PAN

The merchant still knows important business data: the order, amount, customer, payment status, card brand, and often the final four digits for display. But the full card number, called the Primary Account Number (PAN), does not need to enter the merchant backend.

The core sentence: the card is entered into a payment provider controlled browser component, sent directly to the provider, replaced with a provider side reference, and only that reference is used by the merchant's normal application.

One important honesty note

Payment providers do not publish every internal detail of their production vaults. This article is therefore not a claim about Stripe's private implementation. It is an industry architecture model built from public integration behavior, PCI guidance, browser security rules, tokenization standards, and common high assurance system design.

S02 The problem we are solving

“Encrypt it with HTTPS” is necessary, but it is not the whole solution.

HTTPS protects data on the wire. It does not stop your own frontend, backend, logs, queues, support tools, or compromised scripts from seeing data that your application collected.

Suppose the merchant builds ordinary HTML fields and submits the card number to its own backend. The browser may use HTTPS, so a person sniffing Wi Fi traffic cannot read the request. But the merchant now possesses the PAN and CVC.

HTTPS solves

Protection against passive network eavesdropping and tampering while the browser talks to the intended server.

confidentiality in transitserver authenticationintegrity in transit

HTTPS does not solve

Exposure after the application receives the data, accidental logging, malware inside the page, broad employee access, unsafe backups, or a compromised backend.

application exposurelogs and tracesdatabase breach
If raw card data reaches your server, every system that can receive, process, persist, debug, replay, back up, or observe that request can become part of the card data security problem.

Reducing the number of systems that handle card data can materially reduce risk and compliance scope. It does not mean the merchant has zero PCI responsibility; exact eligibility depends on the integration and validation method.

S03 Vocabulary before architecture

Meet the systems involved in one card payment.

The words are more confusing than the flow. Once the actors are clear, the architecture becomes much easier to reason about.

Customer / Cardholder

The person entering the card and authorizing the purchase.

Merchant

The business selling the product. It owns the checkout page and order system.

PSP / Gateway

A provider such as Stripe like infrastructure that exposes APIs, captures payment details, and orchestrates payment processing.

Acquirer / Processor

The merchant side financial institution or processing layer that connects transactions toward card networks.

Card Network

A scheme such as Visa or Mastercard that routes messages and applies network rules.

Issuer

The bank that issued the customer's card and ultimately approves or declines the authorization.

S05 Browser isolation

The card field looks embedded, but it belongs to another origin.

Hosted fields use one or more cross origin iframes so the browser, not a promise in application code, enforces a boundary between merchant scripts and payment field DOM.

The merchant loads the PSP's JavaScript SDK and asks it to mount a payment component. The SDK inserts an <iframe> whose document is served from a PSP controlled origin. The customer sees a seamless card form, but the sensitive input nodes live in a separate document.

Why the merchant cannot simply read the card field

The browser's Same Origin Policy restricts scripts from one origin from reading or manipulating the DOM of a document from another origin. The merchant can position and style the outer container, but it gets very limited access to the iframe's internal document.

How does the iframe communicate without revealing the PAN?

Cross origin windows can use window.postMessage() for controlled communication. The iframe can report non sensitive state such as “complete,” “invalid expiry,” or “card brand detected.” The parent page should validate the sender origin exactly, validate message shape, and use an explicit target origin when sending messages.

// Conceptual parent page listener
window.addEventListener('message', (event) => {
 if (event.origin !== 'https://secure.psp.example') return;

 const message = validatePaymentFieldMessage(event.data);
 payButton.disabled = !message.complete;
});

// Avoid loose checks such as:
// event.origin.endsWith('psp.example')
// because lookalike domains can defeat naive string matching.
The iframe is a powerful containment boundary, not a magic shield. A malicious merchant page script could still place a fake field over the real iframe, replace the container, or trick the customer. That is why payment page script integrity and tamper detection also matter.
S06 Replace the credential

Tokenization turns “use this card” into “use this provider side reference.”

The merchant receives an opaque identifier that can be sent to the payment provider, stored with a customer, and used in later payment flows without storing the PAN itself.

After the hosted field validates the card details, the provider creates a token, payment method object, or equivalent reference. The exact object name differs by provider and API generation. Conceptually, the result looks like this:

{
 "payment_method_id": "pm_7Hk9...",
 "card": {
 "brand": "visa",
 "last4": "4242",
 "exp_month": 12,
 "exp_year": 2030

// Full PAN: not returned to merchant code
// CVC: not returned and not stored for future use

The merchant backend now uses its secret server side credential to call the PSP. The browser should use only provider approved public or publishable configuration. A secret API key must never be embedded in frontend JavaScript.

What a token helps with

Reduces PAN exposure, supports saved payment experiences, and lets most systems use a non PAN identifier.

What a token is not

It is not automatically harmless. A usable token plus compromised API credentials may still authorize charges, so access control remains essential.

What remains merchant data

Order IDs, customer IDs, amount, currency, status, provider IDs, brand, and masked display information.

The merchant stores the pointer, not the secret behind the pointer. The provider's authorization layer decides which account, merchant, API credential, and operation may use that pointer.
S07 Protect the path

TLS protects the sensitive request while it travels to the provider.

Isolation decides who receives the data. TLS protects the network path between the browser and that receiver. Both layers are required.

PSP iframeowns the sensitive DOM
TLS channelconfidentiality + integrity
PSP edgerouting + filtering
Card data zonetokenize / authorize

Modern payment integrations use HTTPS with strong TLS configurations. In practice, providers commonly enforce TLS 1.2 or 1.3, strong cipher suites, valid certificates, and downgrade resistance. HSTS can tell browsers to use HTTPS for future connections to a domain.

What happens at the provider edge?

The request reaches load balancers, edge proxies, or API gateways. Those components may authenticate the integration, apply rate limits, enforce request size and schema rules, detect abuse, and route sensitive endpoints into a restricted card data environment.

Public / publishable browser identity

Identifies the provider account or integration in client side flows. It is designed to be exposed, but it must not grant unrestricted server operations.

Secret server credential

Used from the merchant backend. It must be stored in a secret manager or protected runtime configuration, scoped where possible, rotated, and never logged or shipped to the browser.

A gateway should not casually log raw request bodies from card endpoints. “We can redact later” is weaker than ensuring sensitive payloads never enter general purpose logging pipelines.
S08 The sensitive core

The card vault concentrates sensitive work into a small, hardened boundary.

Instead of making every payment microservice capable of handling PANs, the provider restricts detokenization and key operations to a dedicated card data environment.

A common high level design uses a token vault: a database maps an opaque token to encrypted card data, while cryptographic keys are managed separately. The exact topology varies, but the security idea is stable: separate data, keys, privileges, and network paths.

Envelope encryption: keys protect other keys

Encrypted card recordThe PAN is encrypted with a data encryption key (DEK), usually using an approved authenticated encryption construction.
Wrapped DEKThe DEK is itself encrypted, or “wrapped”, under a key encryption key (KEK). This makes per tenant or per record key rotation and access control more manageable.
HSM / key boundaryRoot key material and sensitive cryptographic operations are protected by a hardened cryptographic module. FIPS 140 3 is one widely used validation standard for cryptographic modules.

Network segmentation

Only approved paths can reach sensitive services. Ordinary analytics, billing, support, and webhook services should not have detokenization access.

Strong identity

Human and workload access uses strong authentication, least privilege, short lived credentials, approval controls, and auditable actions.

Key separation

Encrypted card records alone should not be enough. Key systems are separately controlled, monitored, rotated, and protected.

Why use an HSM?

A Hardware Security Module is a cryptographic module designed to protect keys and perform sensitive operations inside a controlled boundary. It can enforce roles, authentication, non exportability, audit behavior, and tamper response properties. “Using an HSM” is not a substitute for sound architecture, but it meaningfully raises the difficulty of extracting root key material.

Do not assume that every provider uses exactly “one HSM, one vault database, and AES 256 in this exact sequence.” Those are common architectural patterns, not public proof of any particular provider's private implementation.
S09 The strictest data rule

The CVC is collected for a transaction, then it must not be retained after authorization.

A saved card is represented by a token or payment method reference. The original CVC is not kept so that future transactions can silently replay it.

PAN

The full card number may be retained only within an appropriately protected card data environment, typically encrypted or replaced with a token outside that environment.

CVC / CVV

PCI rules prohibit retaining the card verification code after transaction authorization, even if someone proposes encrypting it for later use.

But doesn't the provider eventually need the actual card number?

Often, yes. If the provider is authorizing with the underlying PAN, a tightly restricted component must resolve the token and make the payment credential available long enough to construct the outbound authorization message. A network token flow may use a scheme issued token and cryptogram instead.

Transient plaintext is a real engineering problem

Encryption at rest does not mean plaintext never exists. A system that must send a PAN to a downstream network may briefly hold it in process memory. High assurance designs minimize that window, minimize copies, isolate the process, avoid core dumps, avoid swapping where appropriate, use carefully designed secret buffer abstractions, and keep observability away from sensitive values.

Do not rely on “the garbage collector will erase it immediately.” Garbage collected runtimes make deterministic memory erasure difficult. The stronger controls are isolation, short lifetime, minimal copying, safe data types, restricted debugging, and preventing sensitive values from entering general purpose memory paths.
S10 Not all tokens are the same

A PSP vault token and a network token solve related but different problems.

Both replace a PAN in parts of the payment flow. Their issuer, scope, lifecycle, and transaction behavior are different.

Network token domain controls

EMV payment tokenization supports restrictions that bind a token to an intended use context, for example, a device, merchant, or presentment mode. A transaction cryptogram can provide fresh proof for a specific transaction. These controls make a stolen token less useful outside its authorized domain.

Where does ISO 8583 fit?

Many traditional payment network and processor connections use ISO 8583 or network specific dialects derived from similar concepts. 3 D Secure is a separate layer

EMV 3 D Secure helps authenticate the cardholder in card not present payments. A low risk payment may complete through a frictionless flow; a higher risk payment can trigger a challenge such as an OTP, banking app approval, or biometric step. 3DS is not tokenization, it is an authentication protocol that can be invoked as part of the payment flow.

S12 The browser is still hostile territory

An iframe blocks direct DOM reading, but e skimming attacks can target the page around it.

Modern payment security must protect not only backend databases but also the scripts, headers, frames, and rendered checkout page seen by the customer.

A compromised analytics script, tag manager, chat widget, CDN dependency, or injected script may not be able to read the cross origin iframe's internal DOM. But it can still manipulate the merchant page.

Fake overlay

An attacker places a pixel perfect fake card form over the legitimate hosted field and forwards the captured data elsewhere.

Script replacement

A compromised dependency changes how the checkout is rendered or silently redirects the user to a malicious flow.

Page tampering

Unauthorized changes alter security relevant headers, form actions, frames, event handlers, or submission behavior.

PCI DSS v4.x requirements 6.4.3 and 11.6.1 focus on payment page script authorization, integrity, inventory, and detection of unauthorized changes. The standard defines security outcomes; a particular implementation technique such as a MutationObserver may help, but it is not the only architecture.

Do not load every marketing and analytics script on the payment page by default. The smallest trusted script set is easier to inventory, protect, monitor, and reason about.
S13 Inside the payment provider

Once inside, the rule becomes: no implicit trust and no accidental copies.

A secure vault is weakened if decrypted data can leak into logs, traces, Kafka topics, support dashboards, crash dumps, or broad internal networks.

Service to service identity

Mutual TLS can authenticate both sides of an internal connection. Workload identity systems and a service mesh can automate certificate issuance and rotation, while authorization policy decides what each service may call.

Application authorization

mTLS answers “which workload is calling?” It does not by itself answer “may this workload de tokenize PANs?” Sensitive operations need explicit, narrow authorization and audit trails.

Useful DLP controls

Defense in depth scanners can detect PAN shaped digit sequences, apply Luhn validation, redact or quarantine suspicious telemetry, and alert security teams. They are a backup control, not permission to log raw requests. Encodings, formatting, false positives, and incomplete context make perfect detection impossible.

The best log redaction is architectural: sensitive values never reach the generic logger. Regex redaction is the safety net beneath that design, not the primary boundary.
S14 Security is not enough

A secure payment that charges twice is still a broken payment system.

Idempotency, state machines, retries, reconciliation, and immutable audit records protect financial correctness when networks time out and clients retry.

Suppose the PSP sent an authorization request, the issuer approved it, but the response was lost. The merchant sees a timeout. Should it retry? Without a stable payment intent and idempotency key, a retry can create a second charge.

Merchant sends payment requestThe request contains an order/payment intent and an idempotency key.

Security boundary versus correctness boundary

Card data security

Hosted fields, tokenization, TLS, vault isolation, key management, no CVC retention, logging controls.

Financial integrity

Idempotency, durable state, double entry ledgering, safe retries, reconciliation, webhooks, audit history.

These concerns interact but should not be mixed. A token prevents the merchant from handling the PAN; it does not by itself prevent duplicate charges. An idempotency key prevents duplicate operations; it does not protect card data.

The final truths

HTTPS protects data during transit; it does not keep data out of the application that collected it.
Hosted fields place sensitive card inputs inside a PSP controlled cross origin iframe.
The browser's Same Origin Policy prevents ordinary merchant scripts from reading the iframe DOM.
The iframe sends PAN, expiry, and CVC directly to the provider over a protected connection.
The merchant receives a token or payment method reference rather than the full PAN.
A token reduces PAN exposure but still needs authorization, scoping, and secret key protection.
The provider confines credential storage and resolution to a segmented card data environment.
Envelope encryption separates encrypted records, data keys, and root key encryption controls.
CVC must not be retained after authorization, even for future recurring payments.
Some authorizations use the underlying PAN; others can use card network issued tokens and cryptograms.
An iframe reduces direct DOM exposure but does not eliminate page overlay, supply chain, or e skimming threats.
Sensitive values must be excluded from logs, traces, queues, crash dumps, analytics, and broad internal services.
Idempotency and payment state machines protect financial correctness; they are separate from tokenization.
The strongest architecture continuously minimizes who can see the credential, how long they can see it, and what they can do with it.
One line holy grail: the merchant embeds a PSP controlled card field; the browser sends card data directly to the PSP; the PSP returns a constrained reference; only the provider's hardened card data boundary can resolve that reference for authorization; every other system uses tokens, masked metadata, and payment state.

Interview ready answer

We should avoid letting raw PAN or CVC touch merchant servers.
The checkout loads the payment provider's SDK, which renders hosted
card fields in a cross origin iframe. The browser's same origin policy
prevents merchant JavaScript from reading the iframe DOM.

The iframe sends card details directly to the PSP over TLS. The PSP
returns a token or payment method ID. Our frontend sends only that ID,
the order ID, and amount to our backend. The backend uses a secret,
scoped API credential and an idempotency key to create or confirm the
payment.

Inside the PSP, only a segmented card data environment can resolve the
token. Card data is encrypted at rest with separated key management,
and sensitive key operations are protected by hardened cryptographic
modules. CVC is not stored after authorization. The PSP then sends the
PAN or a network token through the acquirer/card network to the issuer.

The issuer approves or declines, the result returns to us, and our
normal services store only payment state, provider IDs, and masked card
metadata. We also protect the payment page against e skimming and make
sure PAN never leaks into logs, traces, Kafka, analytics, or support tools.
