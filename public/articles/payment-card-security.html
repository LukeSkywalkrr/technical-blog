00: Prologue

The button that hides a vault.

Jane taps Pay ₹4,999. A moment later, Payment successful.

Between those two events, several things she did not ask for happened. Shopify appeared to receive her card. It did not. The card number appeared to be sent to HDFC Bank. It was, but not directly. And nowhere in Shopify's databases is Jane's card actually stored.

Behind that small Pay button is a system solving a stubborn design problem:

  • Shopify needs a payment, but Shopify should not hold Jane's card.
  • HDFC needs the card to charge it, but HDFC is not on Shopify's website.
  • Jane expects a single form and a single button, all inside the store's page.

Three companies. One page. One moment. Only one of them is allowed to see the sixteen digits, and it is neither Shopify nor HDFC.

It is Stripe, the payment provider sitting invisibly between them. Let us build the system one problem at a time.

Honesty note

Stripe, like all payment providers, does not publish every internal detail of its production vault, HSM configuration, or key management. What follows is an industry architecture model assembled from Stripe's public documentation, PCI DSS guidance, browser security rules, EMVCo tokenization standards, and common high assurance system design, not a claim about Stripe's private implementation. Where the article says "Stripe does X," it means X is publicly documented or observable in the integration. Where it says "the vault does Y," it means Y is the standard pattern in this space.

01: The illusion

It feels like Jane gave Shopify her card.

She typed the digits on Shopify's checkout page. Shopify got them. Simple, and wrong.

When Jane checks out at headphonesshop.myshopify.com, the natural mental model is a straight line: her browser sends the card to Shopify, Shopify asks HDFC, HDFC replies. Something like this:

The naive design, Shopify sits in the middle.

Clean. Except this design breaks a promise we did not know we were making.

Shopify is not one server. It is a running application with a frontend, a backend, request logs, error trackers, analytics, backups, a queue for slow work, thousands of merchants with dashboards that can search production data, and dozens of third party apps that merchants install and that load code into their storefront. Anywhere the card touches, the card can leak.

Even Shopify's honest engineers are a problem. Nobody at Shopify should be one careless log line away from a database of live card numbers belonging to millions of shoppers.

So the real question is not how do we send Jane's card safely. It is:

How do we let Jane enter a card on Shopify's page without Shopify ever holding it?

That question is what Stripe answers, and it is what the rest of this article is about.

02: The first misconception

HTTPS looks like enough. It isn't.

HTTPS protects the wire. It stops at the front door.

The instinct of most engineers is to say: fine, Shopify collects the card and sends it to HDFC over HTTPS. That defeats someone sniffing coffee shop Wi Fi, and the URL bar shows the reassuring lock icon. Problem solved.

HTTPS does solve exactly what it advertises: the network path between Jane's browser and Shopify's server is confidential, authenticated, and tamper evident.

But the moment Shopify's Ruby backend decrypts the request, HTTPS is done. What remains is a payload sitting in application memory that any of the following can see or accidentally record:

  • Rails' request logger, which writes every request to a log file.
  • Whatever middleware runs before the payment handler, auth, rate limits, telemetry, feature flags.
  • Datadog or a similar tracer, which stringifies request bodies for debugging.
  • The error tracker, which uploads the failing request to Sentry or Bugsnag.
  • The Kafka topic that a downstream fulfillment worker reads from.
  • The nightly database backup that copies rows to S3.
  • The admin tool that lets a Shopify support engineer replay a customer's failed order.

None of these are careless, they exist for good reasons. But every one of them is now a place where a card number can live longer than it should.

The stronger statement: if raw card data reaches Shopify's servers, every system inside Shopify that can receive, process, persist, debug, replay, back up, or observe that request has become part of the card security problem.

The realization is that we are asking the wrong question. Better encryption between Jane and Shopify does not help, because Shopify was never supposed to see the card in the first place.

The goal is not protect the card inside Shopify. The goal is keep the card out of Shopify entirely, and hand it to Stripe instead.

Which is a strange goal, because Jane is typing the card on Shopify's page. So who is receiving it, and how did they get in?

03: The cast

Six companies, one payment.

Before the architecture makes sense, the vocabulary has to. This time, with names.

Almost every confusing conversation about payments turns out to be a vocabulary problem in disguise. So, briefly, the players in Jane's ₹4,999 headphone purchase:

The six companies in Jane's payment. Stripe is highlighted because it does the work the rest of this article is about.

Notice that Stripe collapses several of these roles behind one API. Shopify does not call Visa directly and does not maintain its own relationship with an acquiring bank. It calls Stripe, and Stripe handles everything from the card form on the checkout page down to the ISO 8583 message that eventually reaches HDFC. The conceptual roles above stay useful even though Shopify only ever sees one API.

Two of these six companies share Jane's browser tab: Shopify, whose page she is on, and Stripe, whose card form is sitting inside it. Somehow both of them are in the same window, and only Stripe is allowed to see her card digits.

That is the trick worth understanding first.

04: A wall you cannot see

The card field looks embedded. It belongs to somebody else.

The browser draws an invisible line through the checkout page.

Load Jane's Shopify checkout page and she sees a form: card number, expiry, CVC. Type into the card field and it feels like any other input on the store's page. Border, cursor, placeholder text, all styled to match Shopify's theme.

Now peek at the DOM in Chrome's dev tools and something odd shows up. The card input is not really an input. It is an <iframe>, and the document inside that iframe is served from a completely different domain: https://js.stripe.com/, not https://headphonesshop.myshopify.com/.

Shopify styled the outer container. Stripe rendered the actual field.

Two documents. One browser tab. The wall is invisible but strictly enforced.

The wall is called the Same Origin Policy. The rule is short: a script running on one origin cannot read the DOM of a document served from a different origin. "Origin" here means the exact combination of scheme, host, and port. headphonesshop.myshopify.com and js.stripe.com are different origins. The wall is between them.

So even though Shopify's JavaScript is running on the same page as Stripe's card field, Shopify cannot read the value of that input. It cannot copy the DOM. It cannot attach a listener to the field's keydown events. Every one of those operations is refused by Chrome itself, not by a polite request to Stripe.

Concept · Hosted fields

The pattern of embedding a provider owned form inside a merchant page via a cross origin iframe has an industry name: hosted fields. Stripe calls this product Stripe Elements. Braintree Hosted Fields and Adyen Web Components are the same idea from other providers.

Shopify loads Stripe's small SDK (a script from js.stripe.com), calls elements.create('card'), and points it at an empty <div> in the Shopify checkout theme. Stripe's SDK inserts an iframe into that div. Everything sensitive lives inside the iframe.

An iframe is a containment boundary, not a magic shield. A malicious script running on Shopify's page cannot read Stripe's iframe inputs, but it can hide the iframe, cover it with a fake overlay, or redirect the entire checkout. We will return to this in chapter 11.

The wall settles one question and creates another. Shopify cannot read the card. Good. But Shopify still needs to know whether the field is even filled in, so its Pay button can enable. How does Shopify learn that much without learning too much?

05: Talking through the wall

The iframe sends postcards, not the card.

Enough information to run a checkout. Nothing more.

The browser gives cross origin windows a narrow way to talk: window.postMessage(). It is a message passing API, not a shared memory API. Stripe's iframe can send a message out; Shopify's page can listen for it and choose what to do.

Stripe uses postMessage to tell Shopify only the things Shopify needs to know:

  • The card field is complete., so the Pay button can enable.
  • It's a Visa., so Shopify can show the Visa brand logo.
  • The expiry is invalid., so Shopify can show an inline error.

None of these are the card number. None of them are the CVC. They are metadata about a value Shopify is not allowed to see.

Shopify's checkout page, parent window
// Listen for messages from the Stripe iframe.
window.addEventListener('message', (event) => {

  // Rule 1: check the sender's origin, exactly.
  if (event.origin !== 'https://js.stripe.com') return;

  // Rule 2: validate the shape of what you received.
  const msg = validateStripeMessage(event.data);
  if (!msg) return;

  // Rule 3: act on only the fields you expect.
  payButton.disabled = !msg.complete;
  brandLogo.src = brandFor(msg.brand);  // "visa"
});

// Never: event.origin.endsWith('stripe.com')
// The malicious evilstripe.com origin passes that check.

The three rules, exact origin, validated shape, expected fields, are the difference between a safe listener and a hole. The failure mode is easy to picture: an attacker embeds an iframe from jsstripe.com or stripe.evil.com, sends a message that looks legitimate, and Shopify's page happily believes them.

So Shopify now knows Jane has typed something Visa shaped and complete. The Pay button lights up. Jane clicks it. And at that moment, Stripe's iframe does the one thing Shopify cannot do: it takes the Visa card, and sends it directly to Stripe's servers over TLS. Shopify's servers are not on that path at all.

Same origin policy decides who receives Jane's card. TLS protects the path from her browser to Stripe. Both are required.

Seconds later, Stripe's iframe sends Shopify one more message. This one is the point of everything so far.

06: The receipt

What comes back is a claim ticket for something you never touched.

Tokenization, and the smallest object the merchant needs.

The Visa card is now inside Stripe. In exchange, Stripe hands Shopify an opaque identifier, a PaymentMethod ID, in Stripe's naming. It looks like this:

What Stripe returns to Shopify's browser{
  "id": "pm_1PxK9x2eZvKYlo2C5tHm3aVz",
  "object": "payment_method",
  "type": "card",
  "card": {
    "brand":     "visa",
    "last4":     "9012",
    "exp_month": 12,
    "exp_year":  2030,
    "country":   "IN"
  }
}

// The full PAN is not here.
// The CVC is not here, and is not stored.

Shopify's checkout page receives this JSON, hands the PaymentMethod ID to Shopify's backend along with the order details, and the whole thing gets treated like a coat check ticket. Shopify never had the coat; it only has the number on the paper stub. Handing that stub back to Stripe means charge whatever this represents.

Shopify's frontend → Shopify's backendPOST /checkout/complete
{
  "order_id":         "ord_2024_10428",
  "amount":           499900,
  "currency":         "INR",
  "payment_method":   "pm_1PxK9x2eZvKYlo2C5tHm3aVz"
}

Notice what is not in that payload. No PAN. No CVC. No expiry beyond what Shopify needs to display. From this point on, every Shopify system, the order service, the fulfillment queue, the finance database, the analytics warehouse, the support console the merchant uses to look up "why did this order fail," the nightly backup to S3: can traffic in tokens and never once see Jane's card number.

The token has three properties that make it work:

  1. It is opaque. Nothing about pm_1PxK9x2eZv... reveals the Visa card behind it. You cannot decode it. Only Stripe can.
  2. It is scoped. The token is only usable inside the Stripe account that created it, from the secret API key that owns it. Steal the token from Shopify's database, and without Shopify's Stripe secret key (which lives on Shopify's backend, not in the browser), the token is a piece of string.
  3. It is revocable. Stripe can invalidate a token at any time without Jane needing to reissue her Visa card.
The shift in what Shopify stores: instead of a secret (the card), Shopify stores a pointer (the Stripe PaymentMethod ID). Stripe decides what the pointer resolves to, who is allowed to resolve it, and for what.

So Shopify is safe. The token is useless without Shopify's server side Stripe key, and Jane's card itself never left Stripe. The natural next question is: fine, Stripe has the card now. Where does Stripe actually keep it, and who at Stripe gets to see it?

07: Inside the vault

Even inside the provider, the card is kept in the smallest possible room.

Not every service at the payment company is allowed to see a card number.

Imagine designing Stripe's own architecture. Jane's Visa has arrived. You need to store it somewhere so future charges on this PaymentMethod can succeed. The naive answer is: put it in a Postgres table, encrypt the column, done.

The mature answer is stranger, and much more careful. Stripe does not publish the exact topology of its vault, but its public engineering posts, PCI attestations, and industry norms all point to the same three ideas stacked on top of each other.

Idea 1: Separate the record from the key.

The encrypted card record sits in one system. The key that decrypts it sits in another. Stealing the database gets you ciphertext. Stealing the key management system gets you keys with nothing to decrypt. You need both, and they are protected by different controls.

Idea 2: Wrap keys with keys.

The card is not encrypted directly with a master key. It is encrypted with a per record or per tenant data encryption key (DEK). That DEK is itself encrypted, "wrapped", under a longer lived key encryption key (KEK). The KEK never leaves a hardened boundary. Rotating a DEK is a small operation. Rotating a KEK is a rare, well audited one. This layered structure is called envelope encryption.

Idea 3: Put the root key inside hardware that will not release it.

The KEK lives inside a Hardware Security Module (HSM), a cryptographic appliance designed so that keys can be used inside it but not extracted. You can ask the HSM to unwrap a DEK for you; you cannot ask it to hand over the KEK itself. FIPS 140 3 is the widely used standard that describes what "hardened" means here.

Envelope encryption. Each layer protects the one above it.

Reading Jane's Visa back is a small ceremony. A tightly restricted service inside Stripe asks the HSM to unwrap the DEK; the DEK is used to decrypt the PAN record; the plaintext PAN is used to build the outbound authorization message to Visa; and then everything is discarded. No other service at Stripe, not the fraud engine, not the merchant dashboard, not analytics, not the billing pipeline, not the support tool that Shopify's account manager might use, is on that path at all. Network segmentation, workload identity, and explicit authorization all say no to anyone else who asks.

Concept · HSM

A Hardware Security Module is a purpose built cryptographic appliance. It performs signing, decryption, and key wrapping inside a sealed enclosure, enforces role based authentication for those operations, and responds to physical tampering by destroying its keys.

An HSM is not a substitute for good architecture. It is a strong last resort for the specific goal of making key extraction extremely difficult even when the surrounding software fails.

There is one piece of card data even this vault refuses to keep. Not because it cannot, because the rules forbid it. And this is where payment security is at its strictest.

08: The rule that never bends

The CVC is used once, and then it is gone.

Some data cannot be stored, no matter how well you would store it.

The three digits on the back of the card, the CVC, or CVV, or CID, depending on the network, exist to prove that whoever is trying to charge the card has physical access to it. That guarantee is worth exactly nothing if the CVC is written down after the first use.

So the rule is short and absolute: the CVC must not be retained after the transaction is authorized. Not encrypted. Not tokenized. Not "held briefly in Redis just in case the customer wants to retry." The moment the bank replies to the authorization request, the CVC is discarded. This rule is PCI DSS at its most inflexible.

"Encrypt the CVC" is not a valid compromise. The point is not to protect the CVC from a database breach. The point is that a stored CVC allows silent replay of a full card present style credential without the cardholder present. Its value comes precisely from not existing anywhere between transactions.

This creates an obvious question: if the CVC is thrown away, how do saved cards work? How does a subscription service charge you next month without asking you to re enter the CVC?

The answer is that saved cards work with weaker guarantees. A future charge on a saved card uses the stored PAN (or a network token, which we will meet in the next chapter), the stored expiry, and the merchant's demonstration that this is a legitimate recurring or merchant initiated transaction. It cannot present a fresh CVC because there isn't one. The card networks accommodate this with special transaction indicators, this is a recurring charge, not a fresh purchase, and the issuer takes that into account when deciding whether to approve.

The PAN, meanwhile, may live longer. Inside the vault, protected by the layers we just built. Ready to be resolved when the next charge comes in.

But using that PAN turns out to be more interesting than storing it. Because the card networks would prefer, when possible, that even the PAN never leave the vault. And that is where a second kind of token appears.

09: Two kinds of tokens

The word "token" hides two very different things.

A gateway's private reference is not a card network's substitute credential.

Everything so far has used one word, token, for one thing: the opaque identifier the merchant gets back from the payment provider. That is a PSP token, or gateway token. It is a private handle inside a single company's systems.

There is a second animal with the same name, issued by the card networks themselves. It is called a network token, and it is defined by an EMVCo standard called EMV Payment Tokenisation. Visa, Mastercard, and Amex all operate their own token services under it.

The point of a network token is that the PAN itself never needs to travel to the merchant, the acquirer, or the network for a transaction. The merchant holds a network token. The token flows through the payment chain. Only the issuer's tokenization service can map that token back to the underlying account. The real card number stays behind a very different wall.

PSP tokenNetwork token
Issued by The payment provider (Stripe, Adyen, Braintree, …) The card network (Visa TS, Mastercard MDES, …)
Scope Usable inside that provider's account and API Usable across the card network with domain restrictions
Replaces A reference to a stored card in the provider's vault The PAN itself, in messages sent along the payment rails
Transaction proof Provider's server side API credentials A per transaction cryptogram generated with the token
If it leaks Useless without the provider's secret key Useless outside its bound domain, and requires a valid cryptogram
Lifecycle Provider updates when card is reissued (if the network supports it) Network keeps the token stable when the card is reissued, the number under it changes, the token does not

Network tokens have a property that PSP tokens do not: the token itself is bound to a domain of use. A token issued for card on file at a specific merchant will be rejected if it shows up somewhere else. A token bound to a specific device will not work from a different device. And each transaction needs a fresh cryptogram, which prevents replay.

A stolen PSP token gets you nothing without the provider's server side credential. A stolen network token gets you nothing without its bound domain and a valid cryptogram. Both are strong. They just draw the boundary in different places.

So the token that leaves the vault, on its way to the bank, may not be a card number at all. It may be a card network issued substitute with a cryptographic proof attached. But either way, the message now has to travel out of the payment provider, across the financial rails, and into the bank that issued the card. Which is a longer journey than most people realize.

10: The long path to the bank

The message leaves the provider and enters an older world.

Modern JSON goes in. A 1980s message format comes out. It works.

Shopify's backend has called Stripe with a modern REST request: POST /v1/payment_intents/{id}/confirm, with the PaymentMethod ID Jane's browser gave it. Inside Stripe, the PaymentMethod is resolved into either Jane's actual Visa PAN or a Visa issued network token with a cryptogram, wrapped for outbound transmission, and handed to Stripe's connector that speaks to its acquiring bank.

From here the message is no longer JSON. It is ISO 8583, or a Visa specific dialect derived from it (VisaNet uses BASE II format based on ISO 8583 principles), a binary message format designed decades ago for card networks, still used by most of them. It contains a message type, a bitmap indicating which fields are present, and the fields themselves: account credential, amount, currency, merchant category, terminal type, response code, and dozens more.

Down the chain of authorization. Back up the chain, a single approved/declined bit.

HDFC, the bank that gave Jane the Visa, is the only party allowed to decide whether the charge succeeds. It looks at whatever combination of these it can:

  • Is Jane's account active? Not blocked, not expired, not reported stolen.
  • Are the funds or credit available? Does she have ₹4,999 to spare?
  • Does the transaction look normal? Right country, right merchant category (Stripe → e commerce), right velocity.
  • Was Jane authenticated? Did the 3 D Secure OTP HDFC sent to her phone succeed?

The answer is a single response code that flows back up the same chain: HDFC → Visa → Stripe's acquirer → Stripe → Shopify → Jane's screen. Approved. Or declined, insufficient funds. Or declined, requires authentication.

Concept · 3 D Secure

3 D Secure (EMV 3DS) is an authentication protocol layered on top of card payments, primarily for card not present transactions. When a payment looks risky, HDFC can require Jane to authenticate, a HDFC app tap, a biometric prompt, or the classic OTP over SMS. In India, RBI mandates 3DS for most online card payments, which is why Jane always sees an OTP screen.

3DS is not tokenization and is not encryption. It is a way for HDFC to say prove it's really Jane without asking Shopify or Stripe to handle her banking password. When 3DS succeeds, a piece of evidence rides along with the authorization message, and the liability for fraud shifts from Shopify to HDFC.

And that, in Shopify's world, is the end of the payment story. A row in the orders table flips from pending to paid. Jane sees a confirmation page. Shopify emails the merchant. The merchant dispatches the headphones.

Except we are not done. We have protected the vault at Stripe. We have protected the wire between Jane, Shopify, Stripe, Visa, and HDFC. We have protected the message. There are three more places where a real payment system can still fail, and each of them has ended in a headline before.

11: The page is still dangerous

The iframe was not enough. It was never going to be.

An attacker cannot read the card field. They can replace it.

Here is the failure mode. A Shopify merchant installs eleven apps from the Shopify App Store: Google Analytics, a tag manager, a Tidio chat widget, a session replay tool, three A/B testing scripts, a reviews plugin, and three abandoned cart tools nobody remembers adding. Each app loads its own JavaScript into the checkout theme. One of them, often not even the app's own code, but a fourth level dependency of an npm package the app maintainer uses, is compromised.

The compromised script cannot read Stripe's iframe. Same origin policy still holds. But it can do this: draw a pixel perfect fake card form directly over Stripe's real iframe, capture Jane's keystrokes into itself, and quietly send them to an attacker's server. To Jane, everything looks normal. The Pay button even works, the malicious script forwards the card to Stripe's real endpoint after copying it, so Shopify's order still completes.

This is called e skimming, or a Magecart attack after the group that industrialized it. Real Shopify stores have been hit by this. Stripe's vault is untouched. The wire is untouched. Jane's card just never made it to Stripe, because it was intercepted before it left her browser.

The mental correction: Stripe Elements reduces Shopify's exposure to Jane's card, but it does not remove Shopify's responsibility for the checkout page. The page is now the smallest and most valuable attack surface.

PCI DSS v4 responded to this with two requirements aimed squarely at the payment page: 6.4.3 (know which scripts run on the payment page, authorize each one, and confirm their integrity) and 11.6.1 (detect unauthorized changes to headers and script content). Together they push toward a small idea with a large operational tail: you must know, at every moment, exactly what is running on your checkout page.

Concretely, that looks like:

  • An inventory of every script the payment page loads, refreshed continuously.
  • Content Security Policy headers that reject scripts from anywhere else.
  • Subresource Integrity hashes for pinned dependencies, where operationally feasible.
  • A monitor that alerts when a script's content or a security header changes without a release.
  • A ruthless minimum: no marketing tag or analytics library on the checkout page by default.

The last one is the hardest culturally and the biggest lever technically. A checkout page with three scripts is auditable. A checkout page with eleven is a question of when.

So the page is defended. So is the vault. What about the space between them?

12: No trust inside either

The provider's own network is not automatically friendly.

A perfect vault does not help if the log line beside it is a card number.

Inside Stripe, the vault does one job: hold Jane's Visa. Everything else, routing, risk scoring, dispute handling, fee calculation, monthly statements for Shopify, the merchant dashboard, the developer webhooks, the Radar fraud engine, is done by dozens of other services. None of them need her card. Almost all of them touch payment data.

The rule inside Stripe is roughly: no implicit trust and no accidental copies.

No implicit trust means every internal call is authenticated. A service is not trusted because it is on the same VPC. Mutual TLS establishes both sides of every connection; workload identity confirms which specific Stripe service is calling; explicit authorization policy decides whether that service is allowed to invoke this specific operation. Can this workload de tokenize a PAN? is a policy question, not a network question.

No accidental copies means observability is the leak path. This is subtle and worth showing:

Two log lines. One is a disaster.// Dangerous, the whole request body ends up in logs.
logger.info("payment request = {}", requestBody);

// Better, only fields on the allow list.
logger.info(
  "payment attempt order={} pm={} amount={} status={}",
  orderId, paymentMethodId, amount, status
);

// Best, never depend on developers remembering.
// Use types that recognize secrets, structural allow lists,
// central redaction, restricted debug access,
// and data loss prevention scanners as defense in depth.

A payment platform that gets this right treats sensitive values the way a well designed type system treats units. A Pan type has no default toString(). A Cvc type refuses to be serialized. A Card object cannot enter the general purpose structured logger. The redaction regex that scans for card shaped digit sequences in log pipelines exists, but it is the safety net beneath the design, not the design itself.

Which services should ever see a full PAN? Almost none of them.

Every layer we have built so far protects the secret. There is one more class of failure that a perfect secret cannot prevent: charging you twice.

13: A different kind of failure

A secure double charge is still a broken payment.

The card was safe. The customer was charged twice. Both statements are true.

Consider a specific bad moment. Shopify's backend calls Stripe's /v1/payment_intents/{id}/confirm endpoint. Stripe forwards the authorization to Visa. HDFC approves it, and ₹4,999 is placed on hold on Jane's card. And then, before Stripe's response reaches Shopify, a load balancer between them times out. Shopify sees a network error.

Shopify does not know whether the charge succeeded. Retrying might charge Jane twice. Not retrying might mean she paid and the merchant shipped nothing.

None of the machinery we have built so far solves this problem. Envelope encryption does not know whether a request completed. TLS does not distinguish a lost response from a failed one. This is a correctness problem, not a security problem, and it needs its own tools.

An idempotency key prevents a duplicate operation. A token prevents PAN exposure. They are unrelated tools for unrelated failures.

The primary tool is Stripe's Idempotency Key header: a Shopify generated unique identifier attached to each write to Stripe's API. The rule Stripe follows is simple. The first time Stripe sees a given key, it performs the operation, records the outcome, and returns the result. The second, third, and hundredth time it sees the same key, it does not perform the operation again, it returns the stored result of the first attempt.

Shopify → Stripe: a safe retryPOST /v1/payment_intents/pi_1PxK9x.../confirm
Authorization: Bearer sk_live_...
Idempotency-Key: "ord_2024_10428-confirm-a4f2"

// First attempt: HDFC approves, Stripe records success,
//                response is lost between Stripe and Shopify.

// Second attempt: same Idempotency-Key,
//                 Stripe returns the same result. No second charge.

// A conflicting reuse (same key, different amount) is refused
// by Stripe with a 400, not silently accepted.

Sitting under the idempotency key is Stripe's PaymentIntent state machine: requires_payment_method → requires_confirmation → requires_action → processing → succeeded | canceled. Each terminal state is durable and cannot be moved out of. When Shopify is unsure, it does not guess, it queries Stripe with the PaymentIntent ID, or waits for Stripe's payment_intent.succeeded webhook. Stripe's state machine is the single source of truth about what happened to Jane's payment; the network timeout is just noise.

Card data security and financial correctness travel together but do not overlap. Stripe Elements, tokenization, TLS, vault isolation, and log discipline protect the secret. Idempotency keys, durable PaymentIntent state, and webhooks protect the outcome. A serious payment system needs both, and neither implies the other.

That is every part. Time to assemble them.

14: The complete picture

From keystroke to bank response, in one walk.

This is the mental model. Everything above is why each piece exists.

Jane opens headphonesshop.myshopify.com/checkout. Order total: ₹4,999. Shopify's page loads Stripe's SDK from js.stripe.com. The SDK inserts an iframe into a container on the page. Card, expiry, CVC are painted, but they live inside Stripe's iframe document, a different origin from Shopify.

Jane types her Visa: 4532 1094 7823 9012. Stripe's iframe reports back to Shopify through postMessage: Visa, complete, no errors. The Pay button lights up. Jane clicks it.

Stripe's iframe opens a TLS connection to Stripe's tokenization endpoint, sends the card details, and receives back a PaymentMethod ID: pm_1PxK9x2eZvKYlo2C5tHm3aVz, along with the brand (Visa) and last four (9012). Shopify's servers were nowhere on that path.

Shopify's checkout JavaScript hands the PaymentMethod ID, the order ID, the amount, and the currency to Shopify's backend. Shopify's backend recalculates ₹4,999 from the actual cart contents, because a client provided total is not a source of truth, and calls Stripe's API with Shopify's server side secret key and a fresh Idempotency Key.

Inside Stripe, an edge service authenticates Shopify's API key, checks rate limits, and passes the request to the payment orchestrator. Stripe Radar scores it for fraud. If the score demands it, or India's RBI rules require it, a 3 D Secure challenge is invoked, Jane sees an OTP screen from HDFC on her phone and enters six digits. Stripe's vault is asked to resolve the PaymentMethod: it retrieves the encrypted PAN record, asks the HSM to unwrap the DEK, decrypts Jane's PAN into memory just long enough to build the outbound authorization message, and hands it to Stripe's acquirer connector.

The acquirer connector translates the request into ISO 8583 and sends it to Stripe's acquiring bank. The acquirer routes into VisaNet based on the card's BIN (which starts with 4, so Visa). VisaNet delivers the authorization to HDFC. HDFC looks at Jane's account status, available balance, the 3DS evidence, and its own risk signals, and decides approved. The response climbs back up: HDFC → Visa → Stripe's acquirer → Stripe. Stripe marks the PaymentIntent succeeded, sends a payment_intent.succeeded webhook to Shopify, and returns the success response synchronously too.

Shopify's backend updates the order from pending to paid. Jane's screen shows a confirmation page. Somewhere, a fulfillment worker picks up the order and emails the merchant to ship headphones.

Every one of these steps had a reason. Here are the truths that survive the walk:

  1. HTTPS protects data on the wire; it does not keep data out of the application that decrypted it.
  2. Hosted fields put the sensitive card inputs inside a provider owned iframe from a different origin.
  3. The browser's Same Origin Policy prevents ordinary merchant scripts from reading that iframe's DOM.
  4. Cross origin communication uses postMessage, and only carries non sensitive status.
  5. The card travels from the iframe directly to the provider over TLS, bypassing the merchant backend.
  6. The merchant receives a token, a payment method ID, not the PAN.
  7. A token is opaque, scoped to the account, and useless without the server side API credential.
  8. Inside the vault, envelope encryption separates the encrypted record, the data key, and the root key.
  9. The root key lives inside an HSM, which lets keys be used but not extracted.
  10. The CVC is used once and then discarded, it must not be retained after authorization.
  11. A network token replaces the PAN in the payment message itself, bound to a domain, proven by a cryptogram.
  12. The authorization message becomes ISO 8583 as it enters the card networks, and the issuer decides.
  13. The iframe blocks DOM reading but not overlays; the checkout page needs script inventory and integrity monitoring.
  14. Inside the provider, mTLS, workload identity, secret aware types, and central redaction keep secrets out of logs and traces.
  15. Idempotency keys and a durable payment state machine protect the customer from being charged twice by a lost response.
  16. Every layer minimizes who can see the credential, how long they can see it, and what they can do with it.
Shopify embeds a card field it does not own. Jane's browser sends the Visa directly to Stripe. Stripe returns a PaymentMethod ID. Every Shopify system uses that ID, and only Stripe's smallest, most tightly controlled boundary is ever allowed to resolve it against Jane's real card, long enough to build one ISO 8583 message for Visa to route to HDFC.

Jane still typed her card. Shopify still sold her headphones. HDFC still approved ₹4,999. Between those events, sixteen digits were seen by Stripe, Visa, and HDFC, the three companies that had to see them, and by nobody else.

That was the whole design.
