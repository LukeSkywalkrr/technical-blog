S01 The mental model

PostgreSQL is a page-based MVCC storage engine.

The database is not a single magical file. It is a system of backend processes, shared memory, heap pages, index pages, WAL logs, background writers, checkpoints, vacuum, and replication streams.

When you run a query, PostgreSQL does not directly think in terms of “one object” or “one Java class”. It thinks in terms of relations, blocks, pages, tuples, indexes, transaction IDs, snapshots, and WAL records.

ClientSQL request
Backendparse / plan / execute
Shared Memorycached pages + WAL buffers
Diskheap / index / WAL files
The core sentence: PostgreSQL changes heap and index pages in memory, writes WAL for durability, acknowledges commit after WAL is durable, and flushes dirty data pages later.

This article builds from the bottom up. First we understand disk, pages, heap tables and indexes. Then we walk through insert, update, delete, read, checkpoint, crash recovery, vacuum and replication.

S02 Storage layer

What is actually stored on disk?

A PostgreSQL database cluster contains many files. Your table, indexes, WAL, free space maps, visibility maps and TOAST data are separate physical structures.

A table is called a relation. The table’s main data file is the heap. Indexes are separate relation files. PostgreSQL also maintains metadata files that help it find free space and skip visibility checks.

Important correction: heap is not a heap tree

In PostgreSQL, “heap table” means an unordered collection of heap pages. It does not mean the binary heap data structure from DSA. The tree-like structures are generally indexes, especially B-tree indexes.

users table
├── main heap relation file       -- row versions
├── users_pkey index file         -- id -> heap location
├── users_email_idx index file    -- email -> heap location
├── users_fsm                     -- free space map
├── users_vm                      -- visibility map
└── possible TOAST table          -- large out-of-line values
Mental anchor: the table stores row versions; the index stores navigation keys that point to row locations in the heap.
S03 Pages and heap tuples

A table is made of pages. A page contains line pointers and tuples.

PostgreSQL reads and writes fixed-size pages, commonly 8 KB. Rows live as tuples inside those pages.

You may think PostgreSQL stores a table like a spreadsheet. Internally, it stores a relation as pages. Each page contains a header, item identifiers, free space and tuple data.

PageHeaderDataLSN, checksum, free space offsets
Line pointer arrayoffset + length for each tuple

Line pointer #2points to tuple B

How to read this page

The front of the page contains the page header and line pointers. The tuple data normally grows from the end of the page backward. The middle is free space.

A tuple is found using a physical pointer called a CTID.

CTID = (block_number, line_pointer_number)

Example:
(42, 7)

Meaning:
Go to heap page/block 42.
Use line pointer 7.
That pointer tells where the tuple body is.

What is inside a tuple?

A tuple is not only your columns. It also carries MVCC metadata. That metadata lets PostgreSQL decide whether your transaction should see this tuple version.

User columns

id = 101
email = "a@x.com"
name = "Amit"

Hidden metadata

xmin   -- transaction that inserted this version
xmax   -- transaction that deleted/invalidated it
t_ctid -- points to this or newer tuple version
flags  -- tuple state and null bitmap
This hidden tuple metadata is the reason PostgreSQL can have concurrent readers and writers without every read blocking every write.

What about very large values?

Huge values like large text, JSON, byte arrays or documents cannot always fit neatly inside one heap page. PostgreSQL uses TOAST to compress and/or store large values out-of-line in a secondary TOAST table.

S04 Indexes

Indexes are separate structures that point back to heap tuples.

A B-tree index is not the table. It is a navigation structure: key → heap tuple location. It speeds reads but adds write cost.

When you create an index, PostgreSQL creates another physical relation. Its pages are stored on disk and cached in shared buffers like table pages.

CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  email TEXT,
  name TEXT
);

CREATE INDEX users_email_idx ON users(email);

The index finds candidate tuple locations. PostgreSQL often still visits the heap because tuple visibility is not generally stored inside the index.

Good

Faster lookups, range scans, ordering, uniqueness checks, joins and filtering.

Cost

Every insert/update/delete may need to maintain index pages and generate more WAL.

Special case

Index-only scans can skip heap fetches when the visibility map proves the page is all-visible.

Interview line: indexes improve a specific read path, but they increase write amplification, memory pressure, disk usage and vacuum work.
S05 Memory and processes

PostgreSQL uses backend processes plus shared memory.

A backend process executes your query. Shared memory holds cached heap/index pages, WAL buffers, lock tables and transaction state.

Per-backend memory

Each connection is handled by a backend process. That process uses memory for executor state, expression evaluation, sorting, hashing, joining and temporary query work.

work_memtemp_bufferssort/hash memoryexecutor state

Shared memory

Shared memory is visible to multiple backend/background processes. This is where PostgreSQL caches heap/index pages and stages WAL records before writing them.

shared_bufferswal_bufferslock tabletransaction state
Backend #1client A query
Backend #2client B query
shared_buffersheap + index pages
wal_buffersWAL records

Background processes

WAL writer

Helps write WAL buffers to WAL segment files. Commit durability depends on WAL reaching stable storage.

Background writer

Writes dirty shared buffer pages gradually so user backends are less likely to be forced to do sudden writes.

Checkpointer

Creates recovery restart points by making dirty page state durable enough to reduce crash recovery work.

Autovacuum

Cleans dead tuples, updates visibility/free-space metadata and prevents transaction ID wraparound.

Archiver

Archives completed WAL files when archive mode is enabled, commonly used for PITR and backup setups.

WAL sender/receiver

Streams WAL from primary to standby in physical replication.

S06 Write path

What happens when an INSERT commits?

This is the most important flow. The table page does not need to be flushed at commit. WAL does.

INSERT INTO users(id, email, name)
VALUES (101, 'a@x.com', 'Amit');

COMMIT;

Read this carefully: SELECT usually does not write WAL. It may read an index page, then a heap page, then return visible rows. INSERT, UPDATE, and DELETE generate WAL, leave heap and index pages dirty in memory, and those pages become clean after they are flushed to data files.

S07 Update, delete and MVCC

UPDATE usually creates a new tuple version.

PostgreSQL does not simply overwrite the old row. MVCC keeps multiple row versions so old readers and new writers can coexist.

UPDATE users
SET email = 'new@x.com'
WHERE id = 101;

An update normally creates a new version of the tuple and marks the old version as no longer current for future transactions.

What about indexes during UPDATE?

If an indexed column changes, PostgreSQL usually needs new index entries pointing to the new tuple version. If no indexed column changes and the new tuple can fit on the same page, PostgreSQL may use HOT, a heap-only tuple update, which avoids extra index entries.

DELETE

A delete marks the tuple as deleted using transaction metadata. It is not physically removed immediately because older transactions may still need it.

VACUUM later

Once no active transaction can see the dead tuple, vacuum can reclaim the space for future reuse and update visibility metadata.

Interview line: updates and deletes are logical visibility changes first; physical cleanup is a later maintenance process.
S08 Read path

SELECT reads pages, checks visibility, then returns rows.

A read may use a sequential scan, index scan, bitmap heap scan or index-only scan. The final answer must respect the transaction snapshot.

SELECT id, email
FROM users
WHERE email = 'a@x.com';
Planner chooses seq scanCommon when no useful index exists or most of the table will be read anyway.

Index scan does not always mean “no table read.” Often it means “use index to find heap locations, then visit heap to confirm visibility.”

Where does work_mem matter?

Reads can allocate memory for sorting, hash joins, hash aggregation and other executor operations. If the operation does not fit in memory, PostgreSQL can spill to temporary disk files.

S09 Checkpoint and crash recovery

WAL lets PostgreSQL recover without flushing every data page at commit.

A checkpoint creates a restart point. After a crash, PostgreSQL replays WAL from that point to rebuild a consistent database state.

dirty pages
→
checkpoint starts
→
flush required pages
→
record checkpoint

Checkpointing reduces crash recovery time. If there were no checkpoints, PostgreSQL would need to replay a massive amount of WAL. A checkpoint says, in simplified terms: from this point, enough dirty data page state has been written out that recovery can start redo here.

Without WAL

PostgreSQL would need to force all modified heap/index pages to disk before every commit. That would make commits much slower and more random-I/O heavy.

With WAL

PostgreSQL appends WAL records, flushes WAL on commit, and later writes dirty data pages in a controlled background flow.

Full-page writes: why PostgreSQL may log the whole page
After a checkpoint, the first modification to a data page can include a full page image in WAL. This protects against torn-page writes, where a crash leaves the on-disk page half old and half new.
Why checkpoints can affect latency
A checkpoint can cause a wave of dirty page writes. If poorly tuned or if storage is weak, this can create I/O spikes. The general idea is to spread writes smoothly, not force huge bursts.
S10 Vacuum and bloat

MVCC creates old row versions. Vacuum cleans them.

Updates and deletes do not immediately erase old tuples. Autovacuum later reclaims space, maintains visibility information and prevents transaction ID wraparound.

PostgreSQL keeps old tuple versions so readers can use stable snapshots. This is great for concurrency, but it creates dead tuples that must eventually be cleaned.

UPDATE users SET name = 'A' WHERE id = 101;
UPDATE users SET name = 'B' WHERE id = 101;
UPDATE users SET name = 'C' WHERE id = 101;

Possible heap versions:
old version  -> dead
name = A     -> dead
name = B     -> dead
name = C     -> live

Dead tuple cleanup

Vacuum marks space reusable after old snapshots no longer need those tuple versions.

Visibility map

Tracks pages where all tuples are visible to all transactions; helps vacuum and index-only scans.

Free space map

Helps PostgreSQL find pages with reusable space for future inserts/updates.

If vacuum cannot keep up, bloat grows. Bloat means more pages to read, larger indexes, worse cache efficiency, slower scans and higher storage pressure.
S11 Primary / standby replication

Replication is mostly WAL shipping and replay.

The old word “master” is now usually called primary. The standby follows the primary by receiving and replaying WAL.

In physical streaming replication, the primary generates WAL for changes. Standby servers receive those WAL records and replay them to maintain a physical copy of the primary database cluster.

Asynchronous vs synchronous replication

Asynchronous

The primary can acknowledge commit before standby confirms receipt/replay. This is faster, but a primary crash can lose recent committed changes that had not reached the standby.

Synchronous

The primary waits for configured standby acknowledgement before commit returns. This improves durability across machines but increases write latency and can reduce availability if standbys are unhealthy.

Replication lag

Lag means the standby is behind the primary. It may have received WAL but not replayed it yet, or it may not have received it at all. Applications reading from replicas must handle stale reads.

Primary commit order:
A, B, C, D, E

Standby applied:
A, B, C

Replica read may not see D and E yet.

Replication slots

A replication slot tells the primary to retain WAL needed by a standby or consumer. This prevents the primary from deleting WAL that the receiver still needs. But if a slot is abandoned or the standby is down for too long, WAL can pile up and consume disk.

Production line: read replicas help read scaling, but they introduce lag. Do not send read-after-write sensitive traffic to a lagging replica unless you have a consistency strategy.
S12 Production and interview lens

How to reason about PostgreSQL bottlenecks.

When PostgreSQL is slow, map the symptom to CPU, memory, lock contention, disk I/O, WAL pressure, vacuum lag, index design, query plan or replication lag.

CPUbad plans, too many connections, heavy joins, expression evaluation
Memorylow cache hit, work_mem spills, too many active backends
I/Orandom heap reads, checkpoint spikes, temp files, WAL fsync
Lockshot rows, DDL blocking, long transactions, deadlocks

Common production questions

Why is my insert slow?
Check WAL fsync latency, number of indexes, foreign key checks, triggers, lock waits, checkpoint pressure and storage throughput.
Why is my update slow?
Check whether indexed columns are changing, whether HOT updates are possible, row lock contention, dead tuple bloat, autovacuum lag and index bloat.
Why is my read slow even with an index?
The index may not match the predicate, selectivity may be poor, heap fetches may be expensive, visibility map may not allow index-only scan, or the planner may choose a different path due to statistics.
Why is replica read inconsistent?
The standby is behind. WAL may not have been received or replayed. Use primary reads for read-after-write, wait for replay position, or design with eventual consistency.

The final truths

Tables are heap files made of fixed-size pages, commonly 8 KB.
A heap tuple contains both user columns and MVCC metadata.
Indexes are separate disk-backed structures, usually B-trees.
Index leaf entries point to heap tuple locations.
Both heap and index pages are cached in shared_buffers.
Writes modify memory pages first and mark them dirty.
WAL is flushed at commit; dirty data pages can be flushed later.
Crash recovery starts from a checkpoint and replays WAL forward.
UPDATE usually creates a new tuple version under MVCC.
Autovacuum is essential to clean dead tuples and prevent bloat.
Replication streams WAL from primary to standby and can be async or sync.
Read replicas scale reads but introduce replica lag and stale-read risk.
One-line holy grail: PostgreSQL is a page-based MVCC engine where heap and index pages are cached and modified in memory, WAL is flushed first for durability, dirty pages are written later, and recovery/replication both depend on replaying WAL.
