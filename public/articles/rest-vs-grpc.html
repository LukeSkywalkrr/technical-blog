REST vs gRPC: Why the Protocol You Choose Defines the System You Build

From the birth of HTTP/2 to proto buffers, streaming, and where gRPC still can't reach — a practical guide for engineers choosing their service communication layer.

Part 1

HTTP/1.1 was brilliant — and then the web outgrew it

In 1991, Tim Berners-Lee created HTTP/0.9: one request, one response, connection closed. By 1997, HTTP/1.1 arrived with persistent connections and pipelining. For web pages of that era, it was more than enough.

But modern applications aren't 1997 web pages. A single load of a complex UI might trigger 50–100 parallel asset requests — scripts, styles, fonts, API calls. HTTP/1.1 had a structural problem that no amount of engineering could fully patch.

Head-of-line blocking. In HTTP/1.1, each connection can only handle one request at a time. A slow response at the front of the queue blocks everything behind it — even requests that could have completed instantly.

Engineers worked around this with hacks: domain sharding (open more connections), asset bundling (fewer requests), inlining CSS. These solutions added complexity without solving the root cause.

Part 2

HTTP/2 — the protocol rewrite that enabled everything after it

Google's internal SPDY project in 2009 proved the concept. By 2015, HTTP/2 was standardized. It didn't change the semantics of HTTP — your verbs, status codes, and headers still work the same way. What it changed was the wire format.

Feature 1
Multiplexing over a single connection
Multiple requests and responses travel simultaneously over one TCP connection via independent "streams." No more blocking. Request C doesn't wait for Request A.
Feature 2
Header compression (HPACK)
HTTP headers are repetitive — the same Accept, Authorization, and Content-Type are sent on every request. HTTP/2 compresses headers using a shared lookup table, cutting overhead dramatically on chatty APIs.
Feature 3
Binary framing layer
HTTP/1.1 is text-based, parsed character by character. HTTP/2 speaks binary — structured frames that machines parse faster and with fewer errors. This is the foundation gRPC builds on.
Feature 4
Server push
Servers can proactively send resources a client will need before it asks. Less useful in practice than hoped, but it opened the mental model of bidirectional communication.
The key insight: HTTP/2 didn't change what you request — it changed how those requests travel. This binary, multiplexed transport is exactly what gRPC needed to become practical.
Part 3

How gRPC works — the big picture

gRPC is Google's open-source RPC framework, released in 2016. RPC stands for Remote Procedure Call — the idea that calling a function on another service should feel as natural as calling a local function in your own code.

It sits on top of HTTP/2 and uses Protocol Buffers (protobuf) as its interface definition language and wire format. Here's the architecture:

→

Your logic runs

The contract between client and server is defined in a .proto file. The gRPC toolchain generates client stubs and server skeletons in your language of choice — Java, Go, Python, and more. You never write serialization logic by hand.

Why protobuf instead of JSON? Protobuf messages are typed, versioned, and binary-encoded. They're 3–5× smaller than equivalent JSON and significantly faster to serialize. The trade-off: you lose human readability and need tooling to inspect payloads.

gRPC supports four communication patterns — a major advantage over REST's request-response model:

Unary
  • One request, one response
  • Same pattern as REST
  • Most common starting point
Server streaming
  • One request, many responses
  • Live feeds, log tailing
  • Server pushes until done
Client streaming
  • Many requests, one response
  • File uploads, batch sends
  • Server waits, then responds
Bidirectional streaming
  • Many requests, many responses
  • Chat, real-time collaboration
  • Full duplex on one connection
Part 4

Same call, two protocols — in Java

Let's make this concrete. Two microservices: OrderService calls ProductService to fetch a product by ID. Here's how each protocol handles it.

Scenario: OrderService needs product details to complete an order. It calls ProductService with a product ID and expects name, price, and availability back.

REST — ProductService (Spring Boot)

REST Controller
@RestController
@RequestMapping("/api/products")
public class ProductController {

    @GetMapping("/{id}")
    public ResponseEntity<ProductDto> getProduct(@PathVariable Long id) {
        Product product = productService.findById(id);
        return ResponseEntity.ok(ProductDto.from(product));
    }
}

REST — OrderService calling ProductService

REST Client (WebClient)
public ProductDto fetchProduct(Long id) {
    return webClient.get()
        .uri("/api/products/" + id)
        .retrieve()
        .bodyToMono(ProductDto.class)
        .block();
    // Serialization, HTTP verbs, URL construction — all manual
}

Now the same thing in gRPC. First, define the contract in a .proto file:

product.proto
syntax = "proto3";

message ProductRequest {
    int64 id = 1;
}

message ProductResponse {
    int64  id           = 1;
    string name         = 2;
    double price        = 3;
    bool   available    = 4;
}

service ProductService {
    rpc GetProduct (ProductRequest) returns (ProductResponse);
}

gRPC — ProductService server implementation

gRPC Server
public class ProductGrpcService
        extends ProductServiceGrpc.ProductServiceImplBase {

    @Override
    public void getProduct(ProductRequest req,
        StreamObserver<ProductResponse> observer) {

        Product p = productRepo.findById(req.getId());

        observer.onNext(ProductResponse.newBuilder()
            .setId(p.getId())
            .setName(p.getName())
            .setPrice(p.getPrice())
            .setAvailable(p.isAvailable())
            .build());

        observer.onCompleted();
    }
}

gRPC — OrderService calling ProductService

gRPC Client
ManagedChannel channel = ManagedChannelBuilder
    .forAddress("product-service", 9090)
    .usePlaintext()
    .build();

ProductServiceGrpc.ProductServiceBlockingStub stub =
    ProductServiceGrpc.newBlockingStub(channel);

ProductResponse response = stub.getProduct(
    ProductRequest.newBuilder().setId(42L).build()
);
// Type-safe, no JSON, no URL construction, no manual parsing
What you gain with gRPC: type safety enforced at compile time, generated client/server code, binary serialization, and automatic support for streaming — all from a single .proto file.
Part 5

Why gRPC doesn't work in the browser — and what gRPC-Web changes

Here's the uncomfortable truth: despite running on HTTP/2, gRPC cannot be called directly from a browser. Not yet. This trips up many engineers who assume "HTTP/2 in the browser" means "gRPC in the browser."

The reason is precise: gRPC uses HTTP/2 trailers — headers sent after the response body — to communicate status codes and metadata. Browsers expose HTTP through the Fetch API and XMLHttpRequest, and neither gives JavaScript access to HTTP/2 trailers. The browser's network stack hides them.

The gap: gRPC status (success, error code, error message) arrives in trailers. JavaScript in browsers cannot read trailers. The protocol physically cannot complete a gRPC handshake from a browser context.

For years, the workaround was a proxy layer: Envoy or grpc-gateway sitting between the browser and the gRPC service, translating HTTP/1.1 JSON from the browser into gRPC on the backend. It works, but adds latency and operational complexity.

gRPC-Web architecture
Browser (JS)
→
gRPC-Web protocol
→
Envoy / proxy
→
gRPC service

gRPC-Web is Google's official solution: a modified protocol that wraps gRPC frames into HTTP/1.1-compatible responses, moving trailers into the response body so browsers can read them. You still need a proxy, but the proxy speaks gRPC-Web natively, and your browser client uses a generated gRPC-Web stub.

The emerging update — Connect protocol: New

Buf's Connect protocol (2022) takes this further. Connect services are compatible with gRPC, gRPC-Web, and plain HTTP/1.1 JSON — simultaneously, with no proxy required. A Connect server can be called from a browser with fetch(), from a mobile app with gRPC, and from another service with standard gRPC. This is the direction the ecosystem is moving.
Part 6

Where to use each — and the performance reality

Neither REST nor gRPC wins universally. The right answer depends on who the clients are, how chatty the communication is, and how much you value developer ergonomics versus raw performance.

Public APIs
REST's JSON is universally readable, curl-testable, and requires no tooling.
Browser-first apps
Until Connect is universal, REST is the path of least resistance for web UIs.
CRUD services
Simple resource operations map naturally to HTTP verbs. REST is the obvious fit.
Internal microservices
Service-to-service calls benefit from type safety, generated clients, and binary speed.
Real-time streaming
Bidirectional streaming is first-class in gRPC. No polling, no WebSocket workarounds.
Mobile backends
Smaller payloads matter on constrained connections. Protobuf saves bandwidth.

The pragmatic answer: Use REST for anything public-facing or browser-driven. Use gRPC for internal service-to-service calls where you control both ends. As Connect matures, the browser limitation shrinks — watch that space.

The bottom line

HTTP/1.1 gave us the web. HTTP/2 gave us the transport gRPC needed. gRPC gave us type-safe, high-performance, multi-pattern RPC. REST remained dominant at the edge because of human readability and browser compatibility. gRPC-Web closed some of that gap, and Connect is closing more.

Understanding the layers — transport protocol → serialization format → API style → client constraints — is what lets you make the right call on each boundary in your system, rather than applying one hammer everywhere.
