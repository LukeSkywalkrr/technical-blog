§ 01

Requirements and the Consistency Choice

Define what must be fast, durable, and allowed to lag

Users upload, download, synchronize, and share files from many devices. Uploads can reach 50 GB, interrupted transfers must resume, and another device should notice a change within seconds.

The system favors availability with eventual consistency between devices. A laptop may briefly show version 4 while a phone has committed version 5. It is not acceptable to expose a half uploaded file or silently overwrite concurrent work.

Upload and downloadAutomatic syncSharing and ACLsResume after failure
RULE
Consistency boundary: metadata can propagate later, but any version marked ready must point to a complete, readable object.
§ 02

High Level Architecture

Keep large file bytes away from application servers

The most important boundary is between the control plane and the data plane. The control plane authenticates users, stores names and versions, checks permissions, and issues temporary signed URLs. The data plane moves bytes directly through object storage and a CDN.

§ 03

The Core Data Model

One stable file identity, many temporary transfer records

One row in t_file represents one logical file. Updating a file does not change its identity. It changes the version and the object key that the row currently points to.

§ 04

Resumable Multipart Uploads

Parallel parts make a 50 GB transfer practical

The client first asks the File Service to create an upload. For a small file, the service returns one presigned URL. For a large file, it creates an S3 multipart upload and returns a URL for each part.

1Client creates an upload session with name, size, and type
↓
2File Service creates a file in UPLOADING state
↓
3Service creates an S3 multipart upload and signs part URLs
↓
4Client uploads parts directly and records every ETag receipt
↓
5Service verifies ListParts, completes S3, then marks the file READY

An ETag is treated as S3's receipt for one uploaded part. During completion, the backend sends S3 the ordered part numbers and ETags. It verifies with ListParts instead of trusting only the client's report.

Resume state
upload U100   s3_upload_id ABC123
part 1        uploaded   etag A
part 2        uploaded   etag B
part 3        missing    request a fresh URL
part 4        missing    request a fresh URL
WIN
A retry uploads only missing parts. Successful parts remain associated with the same S3 upload ID.
§ 05

Making a New Version Visible Safely

Object storage and the metadata database cannot share one transaction

Suppose version 4 is visible while version 5 uploads. Keep the database pointer on version 4 until S3 confirms the complete version 5 object. Then update the metadata row in one database transaction.

CURRENTfiles/F100/version4

Version 4 stays readable during upload.

→
AFTER S3 SUCCESSfiles/F100/version5

The metadata pointer moves atomically.

If storage succeeds but the database update fails, users still see the valid old version. A finalization retry or reconciliation worker can discover and commit the orphaned object. Cleanup can remove it later if it never becomes referenced.

ORDER
Upload object first, update pointer second. Reversing this order can expose a version whose bytes do not exist.
§ 06

The Download and Sharing Path

Authorize first, then issue temporary access

The client requests a download for a stable file ID. The File Service authenticates the user, checks ownership or the access list, and returns a short lived S3 or CDN URL. The application does not stream the file.

1GET file download request
↓
2Authenticate user and evaluate owner or ACL permission
↓
3Issue a signed CDN URL for the current object key
↓
4Nearest edge serves the bytes, S3 handles a cache miss
CACHE
Cache stable metadata and downloaded bytes. Do not treat an expiring signed URL as permanent file metadata.
§ 07

The Local Sync Agent

Turn operating system events into durable cloud work

A desktop agent watches a designated folder with Windows notifications, macOS FSEvents, or Linux inotify. Create, modify, rename, move, and delete events enter a local durable queue.

One save can emit several events, so the agent briefly debounces them. It uses size and modification time as cheap signals, with a content hash when stronger verification is worth the cost.

Local fieldWhy it exists
file_idConnects a local path to the stable cloud identity
last_synced_versionStates which cloud version the edit was based on

size + modified_timeCheap change detection and restart reconciliation
BACKUP
Filesystem events provide speed. A periodic scan provides correctness when events were missed or the app was closed.
§ 08

Discovering Cloud Changes

Notifications reduce latency, cursors provide correctness

A timestamp query is a simple first design, but an ordered change cursor is safer at scale. Each user or workspace receives a sequence of created, updated, and deleted records. Deletions remain as tombstones so devices that were offline can still observe them.

Change stream
501  F100  UPDATED  version 5
502  F200  DELETED
503  F300  CREATED  version 1

GET /files/changes?cursor=500
next_cursor = 503

SSE or long polling tells a connected device that new state may exist. The device then calls the changes API from its last cursor. Periodic polling remains a fallback.

KEY
The notification is only a wakeup signal. The durable change log is the source of truth.

An offline device does not need every intermediate version. It can collapse several updates for the same file to the latest effective state. If the final record is a deletion, it deletes the local file.

§ 09

Version Checks and Conflicted Copies

Server assigned versions expose concurrent edits

Device A and Device B both edit version 4. Device A commits first, so the server assigns version 5. When Device B uploads with base_version = 4, the server compares it with current version 5 and detects a conflict.

Simple policy

Last commit wins

Use server commit order, never device clocks. This is simple but can hide a user's work.

Safer policy

Conflicted copy

Preserve both versions and create a second file named for the conflicting device.

CLOCKS
Device timestamps are unreliable because clocks drift. The server owns both version assignment and commit order.
§ 10

From Full Uploads to Delta Sync

Multipart upload is not persistent chunk storage

The baseline design uploads a complete immutable object for every version. S3 multipart parts only help transfer that object. After completion, they are not independently replaceable blocks.

Advanced delta sync stores reusable chunks and an ordered manifest. When one region changes, the client uploads only the new chunks and the new version references both old and new content.

ABCDE
ABXCDE

Fixed size boundaries shift after an insertion and can make every later chunk look new. Content defined chunking uses a rolling hash so boundaries usually realign after the edited region.

Chunk hashesFile manifestsDeduplicationReference countsGarbage collection
TRADE
Delta sync saves bandwidth but creates a new storage architecture. Earn that complexity only when large files change frequently in small regions.
§ 11

Reliability, Security, and Cleanup

Design every partial failure as a resumable state

FailureResponse
Part upload failsRetry only that part with bounded backoff
Signed URL expiresAuthorize again and issue a fresh URL for the same part
Finalize called twiceReturn the existing result through an idempotent completion API
S3 succeeds, DB failsKeep the old version visible and reconcile the orphan object
Device misses SSECatch up from the durable cursor or periodic poll
Delete requestedWrite a tombstone first, remove bytes asynchronously

All network paths use TLS. Object storage uses encryption at rest. Signed URLs are short lived bearer credentials and are issued only after ownership or sharing permission checks.

DELETE
Logical deletion must happen before physical cleanup. Otherwise an offline device can never discover that the file disappeared.

Final Interview Summary

Separate metadata from file bytes. File, sharing, and sync services own identities, versions, permissions, and signed URLs. Clients send large objects directly to S3 and download through S3 or a CDN.

Make transfers resumable. Large files use client side multipart upload. The backend records the S3 upload ID, part numbers, and ETags, verifies with S3, and makes a version visible only after object completion succeeds.

Make synchronization recoverable. A local agent watches the filesystem and queues edits with their base version. The server assigns new versions and preserves conflicts. SSE or long polling wakes remote devices, while a durable change cursor and deletion tombstones guarantee catch up.

Keep delta sync separate. Multipart parts are temporary. True bandwidth efficient sync requires persistent content chunks, manifests, rolling hash boundaries, reference counting, and garbage collection.
