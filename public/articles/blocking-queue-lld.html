    How we build a queue
    that knows when to wait.

    A normal queue stores data. A blocking queue coordinates people competing to produce and consume it. The hard part is not FIFO. The hard part is putting threads to sleep without losing work — and waking exactly the right ones.

    Read time18 minutes
    Core designCircular buffer + lock + two conditions
    TargetSDE-2 / FAANG-style machine coding
    00 — The interview

    The problem sounds smaller than it is.

    “Implement a queue” is not the question.

    The interviewer gives us one shared queue. Multiple producer threads will add items. Multiple consumer threads will remove them. The queue has a fixed capacity.

    At first, this sounds like a wrapper around ArrayDeque. Add at the back. Remove from the front. Done.

    Then the constraints arrive:

    • If the queue is full, a producer must wait.
    • If the queue is empty, a consumer must wait.
    • No thread may corrupt the queue.
    • No thread may burn CPU repeatedly checking the same condition.
    • When the state changes, the correct waiting thread must wake up.
    The queue is no longer only a data structure. It is a coordination point between threads.
    INTERVIEW CONTRACT

    Design and implement a generic, bounded, thread-safe FIFO blocking queue in Java.

    • put(E) blocks when full.
    • take() blocks when empty.
    • Multiple producers and consumers are supported.
    • Do not use Java's existing BlockingQueue.
    • Target O(1) insertion and removal.
    01 — The naive queue

    A normal queue solves storage, not coordination.

    FIFO is easy. Concurrent waiting is the real design.

    Start with the simplest possible implementation:

    NaiveQueue.java
    class NaiveQueue<E> {
    Queue<E> queue = new ArrayDeque<>();
    int capacity;

    void put(E item) {
        queue.add(item);
    }

    E take() {
        return queue.poll();
    }
}

    This breaks in two ways.

    First, ArrayDeque is not thread-safe. Two threads may mutate its internal state at the same time.

    Second, there is no useful behavior at the boundaries. When the queue is full, what should a producer do? When it is empty, what should a consumer do?

    What we must avoid
    • Return failure immediately when the contract says “wait”.
    • Keep looping and checking capacity.
    • Sleep for arbitrary time and retry.
    • Let multiple threads mutate the queue without a lock.
    What the queue should do
    • Protect shared state atomically.
    • Suspend producers when full.
    • Suspend consumers when empty.
    • Wake them when their required condition becomes true.
    02 — Backpressure

    When producers outrun consumers.

    A bounded queue turns overload into waiting.

    Imagine a queue with capacity four. Producers are faster than consumers. The queue fills:

    Interactive queue — capacity 4
    0
    1
    2
    3
    Queue is empty. A consumer calling take() would wait on notEmpty.

    Once all four positions are occupied, another producer cannot safely insert. But dropping the item may violate the API contract, and busy-waiting wastes CPU.

    The producer should become dormant until a consumer creates space. This is backpressure: the queue slows producers down instead of allowing unbounded memory growth.

    A bounded blocking queue converts “the consumer is behind” into a controlled wait.
    03 — One lock

    Protect the state as one transaction.

    Queue contents, indexes, and count must move together.

    Our queue has shared mutable state:

    • the stored items;
    • the index of the next item to remove;
    • the index of the next position to insert;
    • the number of elements currently present.

    These values form one invariant. If we add an item, we must update the array, move the tail, and increment the count as one atomic operation.

    One lock protects the complete queue invariant, not just the underlying container.

    The standard pattern is:

    Lock pattern
    lock.lock();
try {
    // read or modify shared queue state
} finally {
    lock.unlock();
}
    Why lock before try?The lock is acquired first. The finally block guarantees release after a normal return or an exception. A return inside try still executes finally before the value leaves the method.
    04 — Conditions

    Threads are waiting for different truths.

    One condition belongs to consumers. The other belongs to producers.

    A lock answers: Who may touch the queue right now?

    A condition answers: What must become true before this thread can continue?

    Two conditions prevent producers from waking producers and consumers from waking consumers.

    The producer waits on notFull, but after inserting it signals notEmpty.

    The consumer waits on notEmpty, but after removing it signals notFull.

    Condition direction
    // Producer
while (count == capacity) {
    notFull.await();
}
enqueue(item);
notEmpty.signal();

// Consumer
while (count == 0) {
    notEmpty.await();
}
E item = dequeue();
notFull.signal();
    05 — The subtle bug

    Why the condition belongs inside a while loop.

    The condition was true before sleeping. It may not be true after waking.

    While a thread owns the lock, no other correctly implemented queue operation can change the queue. So the first check is trustworthy.

    But await() changes the situation. It atomically releases the lock and suspends the thread.

    A signal means “try again,” not “the condition is guaranteed forever.”

    The awakened thread does not receive the lock immediately. It only becomes eligible to compete for it. Another producer may acquire the lock first and fill the newly created slot.

    Java also allows spurious wakeups. Therefore the correct condition pattern is always:

    The condition loop
    while (!requiredCondition()) {
    condition.await();
}
    Precise mental modelThe initial condition check is protected by the lock. The uncertainty begins because await() releases that lock. After waking, the thread must reacquire the lock and validate the condition again.
    06 — The buffer

    A circular array gives predictable O(1) operations.

    Head removes. Tail inserts. Modulo wraps both around.

    We could wrap an ArrayDeque<E>. That is a reasonable first implementation. But for the final interview design, a fixed array makes the bounded nature explicit and avoids node allocation.

    When an index reaches the end, modulo brings it back to zero.

    Generics: where the type parameter goes

    The interface declares <E>. The implementation declares the same <E> and passes it to the interface.

    Generic declaration
    interface BoundedBlockingQueue<E> {
    void put(E item) throws InterruptedException;
    E take() throws InterruptedException;
}

class MyArrayBlockingQueue<E>
        implements BoundedBlockingQueue<E> {
    // ...
}

    Java does not allow new E[capacity]. Therefore the internal storage is an Object[]. Only values of type E can enter through the public methods, so casting back during removal is controlled.

    07 — The core implementation

    Put and take are mirror images.

    Wait on your own condition. Signal the opposite side.

    The public interface

    BoundedBlockingQueue.java
    import java.util.concurrent.TimeUnit;

interface BoundedBlockingQueue<E> {
    void put(E item) throws InterruptedException;
    E take() throws InterruptedException;

    boolean offer(E item);
    E poll();

    boolean offer(E item, long timeout, TimeUnit unit)
            throws InterruptedException;

    E poll(long timeout, TimeUnit unit)
            throws InterruptedException;

    E peek();
    int size();
    int remainingCapacity();
}

    The complete implementation

    MyArrayBlockingQueue.java
    import java.util.Objects;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

class MyArrayBlockingQueue<E>
        implements BoundedBlockingQueue<E> {

    private final Object[] items;
    private final int capacity;

    private int head;
    private int tail;
    private int count;

    private final ReentrantLock lock;
    private final Condition notEmpty;
    private final Condition notFull;

    MyArrayBlockingQueue(int capacity) {
        this(capacity, false);
    }

    MyArrayBlockingQueue(int capacity, boolean fair) {
        if (capacity <= 0) {
            throw new IllegalArgumentException(
                    "capacity must be greater than zero");
        }

        this.capacity = capacity;
        this.items = new Object[capacity];
        this.lock = new ReentrantLock(fair);
        this.notEmpty = lock.newCondition();
        this.notFull = lock.newCondition();
    }

    @Override
    public void put(E item) throws InterruptedException {
        Objects.requireNonNull(item, "null items are not supported");

        lock.lock();
        try {
            while (count == capacity) {
                notFull.await();
            }

            enqueue(item);
        } finally {
            lock.unlock();
        }
    }

    @Override
    public E take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await();
            }

            return dequeue();
        } finally {
            lock.unlock();
        }
    }

    @Override
    public boolean offer(E item) {
        Objects.requireNonNull(item, "null items are not supported");

        lock.lock();
        try {
            if (count == capacity) {
                return false;
            }

            enqueue(item);
            return true;
        } finally {
            lock.unlock();
        }
    }

    @Override
    public E poll() {
        lock.lock();
        try {
            if (count == 0) {
                return null;
            }

            return dequeue();
        } finally {
            lock.unlock();
        }
    }

    @Override
    public boolean offer(
            E item,
            long timeout,
            TimeUnit unit
    ) throws InterruptedException {

        Objects.requireNonNull(item, "null items are not supported");
        Objects.requireNonNull(unit, "TimeUnit cannot be null");

        long nanos = unit.toNanos(timeout);

        lock.lock();
        try {
            while (count == capacity) {
                if (nanos <= 0) {
                    return false;
                }

                nanos = notFull.awaitNanos(nanos);
            }

            enqueue(item);
            return true;
        } finally {
            lock.unlock();
        }
    }

    @Override
    public E poll(
            long timeout,
            TimeUnit unit
    ) throws InterruptedException {

        Objects.requireNonNull(unit, "TimeUnit cannot be null");

        long nanos = unit.toNanos(timeout);

        lock.lock();
        try {
            while (count == 0) {
                if (nanos <= 0) {
                    return null;
                }

                nanos = notEmpty.awaitNanos(nanos);
            }

            return dequeue();
        } finally {
            lock.unlock();
        }
    }

    @Override
    public E peek() {
        lock.lock();
        try {
            return count == 0
                    ? null
                    : elementAt(head);
        } finally {
            lock.unlock();
        }
    }

    @Override
    public int size() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }

    @Override
    public int remainingCapacity() {
        lock.lock();
        try {
            return capacity - count;
        } finally {
            lock.unlock();
        }
    }

    private void enqueue(E item) {
        items[tail] = item;
        tail = (tail + 1) % capacity;
        count++;

        // One new item exists. Wake one consumer.
        notEmpty.signal();
    }

    private E dequeue() {
        E item = elementAt(head);
        items[head] = null; // release reference for GC
        head = (head + 1) % capacity;
        count--;

        // One new slot exists. Wake one producer.
        notFull.signal();
        return item;
    }

    @SuppressWarnings("unchecked")
    private E elementAt(int index) {
        return (E) items[index];
    }
}
    Why helpers?put(), offer(), and timed offer() all reuse enqueue(). The three removal variants reuse dequeue(). The public methods decide whether to wait; the helpers perform the actual state transition and signalling.
    08 — The API families

    The same operation, three different waiting contracts.

    Blocking, immediate, and timed behavior share the same queue mechanics.

    put(E)

    Insert. If full, wait until space exists.

    take()

    Remove. If empty, wait until an item exists.

    offer(E)

    Insert immediately or return false. Never waits.

    poll()

    Remove immediately or return null. Never waits.

    offer(E, timeout)

    Wait for space only until the timeout expires.

    poll(timeout)

    Wait for an item only until the timeout expires.

    peek()

    Read the head item without removing it.

    remainingCapacity()

    Return capacity − count.

    The interesting concurrency logic remains in the same two conditions. The additional APIs mostly choose a different policy when the required condition is currently false.

    Timed waiting must preserve the remaining budget

    Timed wait
    long nanos = unit.toNanos(timeout);

while (count == capacity) {
    if (nanos <= 0) {
        return false;
    }

    nanos = notFull.awaitNanos(nanos);
}

    awaitNanos() returns the remaining time. We reuse that value because a wake-up may not make the condition valid, and the total timeout must not restart from the beginning.

    09 — Interruptions

    Waiting threads need a way to be asked to stop.

    Interruption is cooperative cancellation, not forced termination.

    A producer may be sleeping because the queue is full. A consumer may be sleeping because the queue is empty. During application shutdown, we may not want them to wait forever.

    Another thread can call:

    Cancellation request
    workerThread.interrupt();

    This does not forcibly kill the worker. It records an interrupt request. Interruptible waiting operations such as Condition.await() respond by throwing InterruptedException.

    The queue propagates interruption. The worker or application decides whether to stop, retry, or clean up.

    That is why blocking methods declare:

    Propagate interruption
    public void put(E item)
        throws InterruptedException {
    // await() may throw; let the caller decide
}

    The caller handles it:

    Worker policy
    try {
    queue.put(task);
} catch (InterruptedException exception) {
    // Restore the interrupt flag for higher-level code.
    Thread.currentThread().interrupt();
}
    Do not swallow interruptionAn empty catch block turns a cancellation request into nothing. The worker continues as though nobody asked it to stop.

    What about lockInterruptibly()?

    lock.lock() acquires the lock normally. lock.lockInterruptibly() additionally allows interruption while the thread is waiting to acquire the lock itself.

    For the first interview implementation, normal lock() plus interruptible await() is easier to explain. Mention lockInterruptibly() as an extension when stronger cancellation responsiveness is required.

    10 — Signal or signalAll?

    Wake only the amount of work that became possible.

    One insertion creates one item. One removal creates one slot.

    Suppose ten consumers are waiting and one producer inserts one item.

    Calling signalAll() wakes all ten. They all compete for the lock. One consumes the item. The other nine discover an empty queue and return to sleep.

    With two dedicated conditions, normal queue operations can use:

    Targeted signalling
    // One item became available.
notEmpty.signal();

// One slot became available.
notFull.signal();

    This does not reduce the system to one producer and one consumer. Every successful enqueue signals another consumer; every successful dequeue signals another producer. Consumers process work outside the queue lock, so many consumers can execute tasks concurrently.

    signalAll()
    • Correct when guarded by while loops.
    • May wake many threads for one item.
    • Creates extra context switching and lock contention.
    • Useful for shutdown or broad state changes.
    signal()
    • Matches one-item / one-slot state changes.
    • Wakes one useful waiter.
    • Works cleanly because conditions are separated.
    • Preferred for normal put and take.
    11 — Runnable proof

    The design should demonstrate waiting, not only compile.

    Use a small capacity so blocking becomes visible.

    The queue does not own producer or consumer threads. They are external clients. A lambda submitted to an ExecutorService is already a Runnable; we do not need separate classes unless we want reusable worker objects.

    BlockingQueueDemo.java
    import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.TimeUnit;

public class BlockingQueueDemo {

    public static void main(String[] args)
            throws InterruptedException {

        BoundedBlockingQueue<Integer> queue =
                new MyArrayBlockingQueue<>(3);

        ExecutorService executor =
                Executors.newFixedThreadPool(4);

        // Two producers
        executor.submit(() -> produce(queue, 0));
        executor.submit(() -> produce(queue, 100));

        // Two consumers
        executor.submit(() -> consume(queue, "C1"));
        executor.submit(() -> consume(queue, "C2"));

        executor.shutdown();
        executor.awaitTermination(30, TimeUnit.SECONDS);
    }

    private static void produce(
            BoundedBlockingQueue<Integer> queue,
            int start
    ) {
        try {
            for (int i = 0; i < 5; i++) {
                int value = start + i;
                queue.put(value);
                System.out.println("put " + value);
            }
        } catch (InterruptedException exception) {
            Thread.currentThread().interrupt();
        }
    }

    private static void consume(
            BoundedBlockingQueue<Integer> queue,
            String name
    ) {
        try {
            for (int i = 0; i < 5; i++) {
                Integer value = queue.take();
                System.out.println(name + " took " + value);

                // Simulate work outside the queue lock.
                Thread.sleep(120);
            }
        } catch (InterruptedException exception) {
            Thread.currentThread().interrupt();
        }
    }
}

    The capacity is deliberately small. Producers will occasionally fill the queue and wait. Consumers will create space and wake them. If consumers start first, they wait on notEmpty until the first item arrives.

    12 — Interview trade-offs

    What makes this an SDE-2 answer.

    The code matters. The reasoning around the code matters more.

    The final design is intentionally simple: one lock, two conditions, one circular array. It is not the highest-throughput queue possible. It is a correct, explainable baseline with clear extension points.

    Array vs linked storage

    • Circular array: fixed memory, cache-friendly, no per-item node allocation.
    • Linked nodes: flexible capacity, but more allocations and GC pressure.

    One lock vs two locks

    • One lock: simpler invariants and easier interview implementation.
    • Separate put/take locks: can allow producer and consumer operations to progress with less contention, but requires more careful coordination.

    Fair vs non-fair lock

    • Non-fair: usually better throughput.
    • Fair: reduces starvation risk by generally honoring wait order, at a throughput cost.

    Shutdown as a follow-up

    A production queue may add a closed flag. Shutdown would reject new items and use signalAll() on both conditions so every waiting thread can wake, inspect the closed state, and terminate.

    Interview positioningImplement put(), take(), the circular buffer, and a demo first. Then add immediate and timed variants. Discuss two-lock optimization and graceful shutdown rather than letting extensions prevent a complete runnable core.
    FINAL DESIGN

    The queue in one screen.

    The implementation is small because every piece has one precise responsibility.

    Producer rulewhile full → await notFull → enqueue → signal notEmpty
    Consumer rulewhile empty → await notEmpty → dequeue → signal notFull
    Generic ruleDeclare <E> on interface and class; use Object[] internally.
    Interrupt rulePropagate InterruptedException; callers restore the interrupt flag.
    Signal ruleUse signal for one-item state changes; signalAll for shutdown/broad changes.
    Complexityput, take, offer, poll, peek, size: O(1). Space: O(capacity).

    A blocking queue is a small system with a surprisingly complete concurrency story: mutual exclusion, condition waiting, backpressure, signalling, interruption, fairness, and bounded memory.

    “A signal does not mean the condition is yours. It means: wake up, reacquire the lock, and check again.”
