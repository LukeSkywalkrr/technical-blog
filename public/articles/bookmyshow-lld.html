BookMyShow LLD: Reserve and Confirm Booking

A timed interview version of BookMyShow that keeps the model small: denormalized show data, ID based APIs, per seat locking, reservation expiry, and a Java demo that proves double booking prevention.

Hard logic Sort selected seats, lock each one, validate availability, reserve, then release locks in reverse order.
Interview move Use a denormalized Show for the runnable version, then explain the production split.

Part 1

The interview problem

BookMyShow looks simple from the outside, but the interview problem is really about ownership of a scarce resource. A user can search shows, select seats, temporarily hold them, and then convert that hold into a booking after payment.

The important part is not movie metadata. The important part is preventing two users from booking the same seat for the same show.

1

Search

User searches shows by city, movie name, language, and genre. Search is read only and should stay simple.

2

Reserve

User selects multiple seats for a show. The system temporarily reserves those seats with an expiry time.

3

Confirm

User confirms after payment. Reserved seats become booked. Expired or cancelled reservations release seats.

90 minute scope: in memory storage, no REST API, no database, no admin CRUD, no real payment gateway, no notification service, and no full theater hierarchy in the first runnable version.
Part 2

The shortcut that saves time

A full domain model looks correct, but it can burn the whole round before the booking logic exists. Movie, theater, screen, show, and seat are all real entities, but you do not need to wire all of them by hand to prove the core behavior.

Do not start with the full model

Manual object wiring distracts from the reserve and confirm flow. It also makes the demo slower to write and harder to debug.

Too much setup
Movie movie = new Movie(...);
Theater theater = new Theater(...);
Screen screen = new Screen(...);
Show show = new Show(movieId, theaterId, screenId);
List<ShowSeat> seats = ...;

Use denormalized Show first

Keep searchable fields directly inside Show. Explain that production would split them into separate repositories.

Interview friendly model
class Show {
    String showId;
    String movieName;
    String theaterName;
    String city;
    String language;
    String genre;
    LocalDateTime showTime;
    Map<String, ShowSeat> seats;
}
Interview script: For the 90 minute runnable implementation, I am denormalizing searchable fields into Show. In production, Movie, Theater, Screen, and Show would be separate entities backed by repositories.
Part 3

Core design map

Before writing classes, anchor the solution with the resource, state machine, API surface, and concurrency rule. This keeps the implementation from drifting into unrelated features.

Question Answer for BookMyShow Why it matters
Category Booking system Use a reserve and confirm pattern.
Core resource ShowSeat The same physical seat has different status per show.
Resource state AVAILABLE, RESERVED, BOOKED Seat availability lives on ShowSeat.
Main maps showId to Show, reservationId to Reservation Direct lookup keeps the code readable under time pressure.

Part 4

Entity model

Keep seat status separate from reservation status. A seat is either available, reserved, or booked. A reservation is pending, confirmed, cancelled, or expired.

Enums
enum SeatStatus {
    AVAILABLE, RESERVED, BOOKED
}

enum ReservationStatus {
    PENDING, CONFIRMED, CANCELLED, EXPIRED
}
Show and ShowSeat
class Show {
    String showId;
    String movieName;
    String theaterName;
    String city;
    String language;
    String genre;
    LocalDateTime showTime;
    Map<String, ShowSeat> seats = new HashMap<>();

    Show(String showId, String movieName, String theaterName, String city,
         String language, String genre, LocalDateTime showTime) {
        this.showId = showId;
        this.movieName = movieName;
        this.theaterName = theaterName;
        this.city = city;
        this.language = language;
        this.genre = genre;
        this.showTime = showTime;
    }
}

class ShowSeat {
    String showId;
    String seatId;
    SeatStatus status = SeatStatus.AVAILABLE;
    String reservedByUserId;
    Instant reservedUntil;
    ReentrantLock lock = new ReentrantLock();

    ShowSeat(String showId, String seatId) {
        this.showId = showId;
        this.seatId = seatId;
    }
}
Reservation and Ticket
class Reservation {
    String reservationId;
    String userId;
    String showId;
    List<ShowSeat> seats;
    Instant expiresAt;
    ReservationStatus status = ReservationStatus.PENDING;

    Reservation(String userId, String showId, List<ShowSeat> seats, Instant expiresAt) {
        this.reservationId = UUID.randomUUID().toString();
        this.userId = userId;
        this.showId = showId;
        this.seats = seats;
        this.expiresAt = expiresAt;
    }

    boolean isExpired() {
        return status == ReservationStatus.PENDING && Instant.now().isAfter(expiresAt);
    }
}

class Ticket {
    String ticketId;
    String reservationId;
    String userId;
    String showId;
    List<String> seatIds;

    Ticket(Reservation r) {
        this.ticketId = UUID.randomUUID().toString();
        this.reservationId = r.reservationId;
        this.userId = r.userId;
        this.showId = r.showId;
        this.seatIds = r.seats.stream().map(s -> s.seatId).toList();
    }
}
Part 5

Booking system service

The service owns the in memory maps and seed data. Auto creating seats is a practical speed move, because it avoids manual Seat object setup in the demo.

Storage and seed data
class BookingSystem {
    Map<String, Show> shows = new HashMap<>();
    Map<String, Reservation> reservations = new HashMap<>();

    BookingSystem() {
        seedDemoData();
    }

    void seedDemoData() {
        addShow("S1", "Avatar", "PVR", "Delhi", "English", "SciFi", 13);
        addShow("S2", "Avatar", "PVR", "Delhi", "English", "SciFi", 18);
        addShow("S3", "Titanic", "INOX", "Delhi", "English", "Romance", 20);
    }

    void addShow(String id, String movie, String theater, String city,
                 String language, String genre, int hour) {
        Show show = new Show(
            id, movie, theater, city, language, genre,
            LocalDateTime.of(2026, 6, 30, hour, 0)
        );

        show.seats = createSeats(id, 10);
        shows.put(id, show);
    }

    Map<String, ShowSeat> createSeats(String showId, int count) {
        Map<String, ShowSeat> map = new HashMap<>();
        for (int i = 1; i <= count; i++) {
            String seatId = "A" + i;
            map.put(seatId, new ShowSeat(showId, seatId));
        }
        return map;
    }
}
Mock data rule: do not manually create Seat s1, s2, and s3. Auto create seats using createSeats(showId, count).
Part 6

Search flow

Search does not need a strategy pattern in the 90 minute implementation. Use chained filters so all provided conditions apply.

SearchFilter and searchShows
class SearchFilter {
    String movieName;
    String city;
    String language;
    String genre;

    SearchFilter(String movieName, String city, String language, String genre) {
        this.movieName = movieName;
        this.city = city;
        this.language = language;
        this.genre = genre;
    }
}

List<Show> searchShows(SearchFilter q) {
    return shows.values().stream()
        .filter(s -> q.movieName == null || s.movieName.equalsIgnoreCase(q.movieName))
        .filter(s -> q.city == null || s.city.equalsIgnoreCase(q.city))
        .filter(s -> q.language == null || s.language.equalsIgnoreCase(q.language))
        .filter(s -> q.genre == null || s.genre.equalsIgnoreCase(q.genre))
        .toList();
}
Common mistake: returning after the first non null filter means city, language, and genre are ignored. Use chained filters so the query behaves like an AND condition.
Part 7

Reserve seats

The reserve method is the core of the design. It receives IDs, converts them to ShowSeat objects, sorts them for stable lock ordering, locks each seat, validates availability, then reserves them.

1
Lookup show

Use map lookup, not stream collect and get first element.

2
Convert IDs to seats

The public API receives seat IDs. The service converts them into ShowSeat objects.

3
Sort and lock

Sort by seatId, acquire locks, and track acquired locks for the finally block.

4
Validate and reserve

If all seats are available, mark them RESERVED and store the Reservation.

Reserve implementation
Show getShowOrThrow(String showId) {
    Show show = shows.get(showId);
    if (show == null) throw new RuntimeException("Show not found: " + showId);
    return show;
}

List<ShowSeat> getSelectedSeats(Show show, List<String> seatIds) {
    List<ShowSeat> selected = new ArrayList<>();

    for (String seatId : seatIds) {
        ShowSeat seat = show.seats.get(seatId);
        if (seat == null) throw new RuntimeException("Invalid seat: " + seatId);
        selected.add(seat);
    }

    selected.sort(Comparator.comparing(s -> s.seatId));
    return selected;
}

Reservation reserveSeats(String userId, String showId, List<String> seatIds) {
    Show show = getShowOrThrow(showId);
    List<ShowSeat> selectedSeats = getSelectedSeats(show, seatIds);
    List<ReentrantLock> acquiredLocks = new ArrayList<>();

    try {
        for (ShowSeat seat : selectedSeats) {
            seat.lock.lock();
            acquiredLocks.add(seat.lock);
        }

        for (ShowSeat seat : selectedSeats) {
            releaseIfExpired(seat);

            if (seat.status != SeatStatus.AVAILABLE) {
                throw new RuntimeException("Seat not available: " + seat.seatId);
            }
        }

        Instant expiry = Instant.now().plusSeconds(300);
        for (ShowSeat seat : selectedSeats) {
            seat.status = SeatStatus.RESERVED;
            seat.reservedByUserId = userId;
            seat.reservedUntil = expiry;
        }

        Reservation reservation = new Reservation(userId, showId, selectedSeats, expiry);
        reservations.put(reservation.reservationId, reservation);
        return reservation;

    } finally {
        for (int i = acquiredLocks.size() - 1; i >= 0; i--) {
            acquiredLocks.get(i).unlock();
        }
    }
}
Part 8

Confirm, cancel, and expiry

A reservation is only a temporary hold. Any path that kills the hold should release seats: cancel, payment failure, expired confirm, or a background sweeper.

Release helper
void releaseSeatsForReservation(Reservation r) {
    for (ShowSeat seat : r.seats) {
        if (seat.status == SeatStatus.RESERVED && r.userId.equals(seat.reservedByUserId)) {
            seat.status = SeatStatus.AVAILABLE;
            seat.reservedByUserId = null;
            seat.reservedUntil = null;
        }
    }
}
Lazy expiry on one seat
void releaseIfExpired(ShowSeat seat) {
    if (seat.status == SeatStatus.RESERVED
        && seat.reservedUntil != null
        && Instant.now().isAfter(seat.reservedUntil)) {

        seat.status = SeatStatus.AVAILABLE;
        seat.reservedByUserId = null;
        seat.reservedUntil = null;
    }
}
Confirm reservation
Ticket confirmReservation(String userId, String reservationId) {
    Reservation r = reservations.get(reservationId);
    if (r == null) throw new RuntimeException("Reservation not found");

    if (!r.userId.equals(userId)) {
        throw new RuntimeException("Invalid user");
    }

    List<ShowSeat> seats = new ArrayList<>(r.seats);
    seats.sort(Comparator.comparing(s -> s.seatId));
    List<ReentrantLock> acquiredLocks = new ArrayList<>();

    try {
        for (ShowSeat seat : seats) {
            seat.lock.lock();
            acquiredLocks.add(seat.lock);
        }

        if (r.isExpired()) {
            releaseSeatsForReservation(r);
            r.status = ReservationStatus.EXPIRED;
            throw new RuntimeException("Reservation expired");
        }

        if (r.status != ReservationStatus.PENDING) {
            throw new RuntimeException("Reservation not pending");
        }

        boolean paymentSuccess = true;

        if (!paymentSuccess) {
            releaseSeatsForReservation(r);
            r.status = ReservationStatus.CANCELLED;
            throw new RuntimeException("Payment failed");
        }

        for (ShowSeat seat : seats) {
            seat.status = SeatStatus.BOOKED;
            seat.reservedByUserId = null;
            seat.reservedUntil = null;
        }

        r.status = ReservationStatus.CONFIRMED;
        return new Ticket(r);

    } finally {
        for (int i = acquiredLocks.size() - 1; i >= 0; i--) {
            acquiredLocks.get(i).unlock();
        }
    }
}
Cancel reservation
void cancelReservation(String userId, String reservationId) {
    Reservation r = reservations.get(reservationId);
    if (r == null) return;

    if (!r.userId.equals(userId)) {
        throw new RuntimeException("Invalid user");
    }

    List<ShowSeat> seats = new ArrayList<>(r.seats);
    seats.sort(Comparator.comparing(s -> s.seatId));
    List<ReentrantLock> acquiredLocks = new ArrayList<>();

    try {
        for (ShowSeat seat : seats) {
            seat.lock.lock();
            acquiredLocks.add(seat.lock);
        }

        if (r.status == ReservationStatus.PENDING) {
            releaseSeatsForReservation(r);
            r.status = ReservationStatus.CANCELLED;
        }

    } finally {
        for (int i = acquiredLocks.size() - 1; i >= 0; i--) {
            acquiredLocks.get(i).unlock();
        }
    }
}
Part 9

Runnable demo

The main method should prove the important behavior fast: search works, U1 reserves A1 and A2, U2 cannot reserve A1, U1 confirms, U2 still cannot reserve the booked seat, and cancellation releases a pending seat.

Main flow
public class BookMyShowRunner {
    public static void main(String[] args) {
        BookingSystem system = new BookingSystem();

        System.out.println("Search Avatar in Delhi:");
        List<Show> shows = system.searchShows(
            new SearchFilter("Avatar", "Delhi", null, null)
        );
        shows.forEach(s -> System.out.println(s.showId + " " + s.movieName + " " + s.showTime));

        Reservation r1 = system.reserveSeats("U1", "S1", List.of("A1", "A2"));
        System.out.println("U1 reserved: " + r1.reservationId);

        try {
            system.reserveSeats("U2", "S1", List.of("A1"));
        } catch (Exception e) {
            System.out.println("Expected failure for U2: " + e.getMessage());
        }

        Ticket ticket = system.confirmReservation("U1", r1.reservationId);
        System.out.println("Ticket confirmed: " + ticket.ticketId + " seats=" + ticket.seatIds);

        try {
            system.reserveSeats("U2", "S1", List.of("A1"));
        } catch (Exception e) {
            System.out.println("Expected booked seat failure: " + e.getMessage());
        }

        Reservation r2 = system.reserveSeats("U2", "S1", List.of("A3"));
        system.cancelReservation("U2", r2.reservationId);
        System.out.println("U2 cancelled A3. A3 should be available again.");
    }
}
Part 10

Mistakes and 90 minute plan

The fastest way to improve this LLD is to avoid over modeling, use maps for direct lookup, and compile repeatedly before the final demo.

Mistake What happened Next time fix
Too much object wiring Time went into Movie, Theater, and Seat setup instead of booking logic. Denormalize Show and auto create seats.
List lookup with get first Finding one show or seat became fragile and noisy. Use maps for ID lookup.
Object based API The main method had to know internal objects. Use reserveSeats(userId, showId, seatIds).
Search early return Only the first non null filter was applied. Use chained AND filters.
One status enum Seat state and reservation state got mixed. Use SeatStatus and ReservationStatus separately.
Locking the whole show Unrelated seats blocked each other. Put a lock inside ShowSeat or lock by showId plus seatId.

2

Implement getShowOrThrow, getSelectedSeats, and search chain filters.

3

4

Stop adding features. Make the demo readable and working.

Compile checkpoints: compile after entities, after reserveSeats, after confirm and cancel, and once more after the final demo.
This article adapts a timed BookMyShow LLD into the Protocol Field Notes style. The implementation uses in memory maps and per seat locks for interview clarity. A production system would add persistent storage, payment integration, reservation cleanup jobs, and distributed locking or transactional seat updates.
