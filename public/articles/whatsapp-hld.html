§ 00

Core Design Choice

Fat chat_server, no Kafka, real time via Pub/Sub

This is the article style design. The central architectural decision is a fat chat_server that owns the full lifecycle of a message , from WebSocket ingress to DB persistence to Pub/Sub publish.

WebSocket connections Client commands Group creation DB writes Offline state Pub/Sub events Push to clients ACK processing
⚡
No Kafka in the main design. Server to server delivery uses a lightweight Pub/Sub layer. Durability comes from the database, not from the message bus.
real time delivery path 
chat_server_1  →  Pub/Sub  →  chat_server_2

Pub/Sub handles real time server to server delivery. Durability is guaranteed by t_message, t_inbox, or cursor tables , not by the transport layer.

Each device opens one WebSocket connection. A user may have many devices , phone, laptop, tablet , each connecting to any chat_server. The load balancer requires no sticky sessions; whichever server receives the connection stores that socket in memory.

§ 02

Core Tables

The database schema powering the whole system

t_user , stores user metadata and durable last seen timestamp. Do not update last_seen_at on every heartbeat; use Redis TTL for live presence and write DB only asynchronously/batched.

t_group , tracks group type (DIRECT or GROUP), metadata, and the critical latest_message_seq counter. For 1:1 chats, a unique direct_key prevents duplicates:

direct_key formula 
direct_key = min(user_1, user_2) + "#" + max(user_1, user_2)
UNIQUE(direct_key)  /* prevents duplicate 1:1 groups */

t_message , the source of truth for every message. Primary key is (group_id, message_seq).

Column Purpose
group_id + message_seqComposite primary key; per group ordering anchor
message_idCan be group_id + "_" + message_seq or generated
client_msg_idSender deduplication key
is_deleted / edited_atSoft delete and edit versioning
attachment_urlPointer to object storage; media is stored separately

Offline delivery , two models:

Option A , Simple
t_inbox
  • One row per undelivered message per device
  • Deleted on client ACK
  • Write amplification: 1 msg × 100 users × 2 devices = 200 rows
  • Good for interview / simple default
Option B , Scalable
t_device_group_cursor
  • Stores last synced seq per device/group
  • Offline messages = seq > last_synced_seq
  • No write amplification on send
  • Preferred for production scale
✅
Interview answer: Use t_inbox for simplicity. Mention t_device_group_cursor as the production scale alternative.

The socket lives only inside the chat_server that received it. No other server can push directly to it; they must route through Pub/Sub.

§ 04

Heartbeat & Last Seen

Presence is approximate by design

Timing configuration 
Heartbeat interval  =  30 seconds
Presence TTL        =  90 seconds   // 3× heartbeat

When heartbeat stops, the server marks the connection dead, removes the local socket, lets the Redis TTL expire, and asynchronously writes last_seen_at to t_user.

💡
Key separation: Memory/Redis TTL = live presence (fast, volatile). DB last_seen_at = durable approximate last seen (slow, permanent).

§ 06

First 1:1 Message

Client may not have a group_id yet

The client sends recipient_user_id + message_text + client_msg_id. The server atomically upserts the direct group:

Server logic 
direct_key = min(sender, recipient) + "#" + max(sender, recipient)

INSERT INTO t_group (group_type=DIRECT, direct_key=?)
ON CONFLICT (direct_key) DO NOTHING
RETURNING group_id   /* or SELECT if row existed */
🔒
The UNIQUE(direct_key) constraint is the correctness guarantee. A Redis lock can be an optimization, but the DB constraint is the real safeguard.
§ 07

Message Sequence Generation

per group atomic counter, not global

Every group has its own increasing sequence. Two concurrent senders in the same group cannot both get 101.

Atomic increment 
UPDATE t_group
SET    latest_message_seq = latest_message_seq + 1
WHERE  group_id = ?
RETURNING latest_message_seq;   /* this becomes message_seq */

/* Then insert the message */
INSERT INTO t_message (group_id, message_seq, sender_id, ...)
VALUES (...);
⚠️
Snowflake IDs are fine for globally unique message_id, but strict per group ordering must use an atomic group level sequence.
§ 08

Sender Idempotency

Handling retries, timeouts, and duplicate sends

The client generates a UUID client_msg_id for every message. The server enforces:

Uniqueness constraint 
UNIQUE(sender_id, sender_device_id, client_msg_id)

If the same client_msg_id arrives again, the server returns the existing message_id/message_seq instead of inserting a duplicate. This covers client retry, network timeout, duplicate send, and gateway retry.

✅
Interview line: Sender idempotency is handled using client_msg_id with a unique constraint.
§ 09

Normal Send Message Flow

The critical rule: persist first, then publish

1
Client sends send_message over WebSocket
│
2
Validate sender is active member of group_id
│
3
Check idempotency via client_msg_id
│
4
Atomically increment t_group.latest_message_seq
│
5
INSERT into t_message ← persist first!
│
6
Write offline delivery state (t_inbox or cursor)
│
7
Publish delivery event to Pub/Sub
│
8
Return SUCCESS + message_id/message_seq to sender
📌
Rule: Persist first, then publish/push. If Pub/Sub fails after the DB write, the message is not lost , it can be recovered on reconnect.

§ 11

Large Group Fanout

Naive fanout doesn't scale , batch and parallelize

📊
500 members × 2 devices = 1,000 device targets. Do not send one call sequentially per device.

Optimized approach: group targets by chat_server → send one batched delivery event per server.

Batching example 
// Naive: 1000 individual sends
for device in devices: send(device)

// Optimized: group by server → 20 batched sends
servers = group_by_server(devices)   // 1000 devices → 20 servers
for server in servers: batch_send(server, devices[server])
§ 12

Receiver ACK & Delivery Idempotency

The server pushes a message; the client ACKs. Both sides use idempotent, monotonic updates.

SERVER → CLIENT
{
  "type":         "new_message",
  "group_id":     "g1",
  "message_seq":  151,
  "message_id":   "g1_151",
  "message_text": "hello"
}
CLIENT → SERVER (ACK)
{
  "type":               "ack_delivered",
  "group_id":           "g1",
  "device_id":          "d1",
  "last_delivered_seq": 151
}
Idempotent update 
last_delivered_seq = MAX(existing_last_delivered_seq, incoming_seq)
last_synced_seq    = MAX(existing_last_synced_seq,    incoming_seq)
✅
Result: at least once delivery + idempotent client handling. Client deduplicates on group_id + message_seq.

§ 14

Reconnect & Sync

On reconnect, the server compares t_group.latest_message_seq against the device's cursor and streams the diff:

Sync example 
t_group.latest_message_seq          =  200
t_device_group_cursor.last_synced_seq  =  150

/* Fetch and send messages 151..200 to device */
SELECT * FROM t_message
WHERE  group_id = ? AND message_seq > 150
ORDER BY message_seq ASC;

§ 16

Read Receipts

CLIENT → SERVER
{
  "type":         "read_ack",
  "group_id":     "g1",
  "user_id":      "u1",
  "last_read_seq": 150
}
Server update 
UPDATE t_user_group_state
SET    last_read_seq = MAX(last_read_seq, 150)
WHERE  user_id = 'u1' AND group_id = 'g1';

Read receipt is user level, not device level. If you read on phone, your laptop chat also shows it as read. For large groups, batch or throttle receipt propagation to avoid flooding every member.

§ 17

Edit & Delete Message

Edit
Update in place
  • Validate sender owns message
  • Check edit window is allowed
  • Update message_text, edited_at, edit_version
  • Publish message_edited event
Delete
Soft delete
  • Set is_deleted = true
  • Null out message_text (tombstone)
  • Record deleted_at
  • Publish message_deleted event

Online clients update the local message by group_id + message_seq. Offline clients receive the latest state during sync , they may never see the original message.

§ 18

Pub/Sub Failure Handling

Redis Pub/Sub is at most once. Dropped events mean missed real time delivery, but no data loss , the message is already in t_message.

1
Client ACKs messages → server tracks cursor/inbox
│
2
Client detects gaps via message_seq numbering
│
3
Client syncs missing messages after reconnect
│
4
Periodic sync acts as final backstop
✅
Interview line: real time Pub/Sub is best effort. Durable delivery comes from DB + ACK + cursor/inbox sync.
§ 19

Message Ordering

No global ordering is needed. Only per group ordering matters, provided by group_id + message_seq.

🔢
If a client receives messages 101, 103 , it knows 102 is missing and can request a sync. The gap detection is built into the sequence design.

  → write inbox/cursor state
  → publish Pub/Sub event

  → client ACK
  → update delivery state (cursor/inbox)

  → update cursor/inbox

Final Interview Summary

In the article style design, chat_server is a fat component. It owns WebSocket connections, writes messages, handles Pub/Sub, pushes messages to connected clients, and processes ACKs.

Messages are stored in t_message using (group_id, message_seq) as the ordering key. message_seq is generated by atomically incrementing t_group.latest_message_seq. Sender retries are handled using client_msg_id with a uniqueness constraint. Receiver duplicates are deduplicated on group_id + message_seq, and cursor/inbox updates use idempotent max updates.

For real time delivery, chat servers use Pub/Sub. If User A is connected to one server and User B to another, the first server writes to DB and publishes a Pub/Sub event; the server owning User B's socket receives it and pushes over WebSocket.

Pub/Sub is not the source of truth. Offline delivery is handled through t_inbox or the more scalable t_device_group_cursor. On reconnect, the device fetches all messages after its last synced sequence. Push notifications are wakeup only. Media lives in object storage; only attachment metadata travels in chat messages.
