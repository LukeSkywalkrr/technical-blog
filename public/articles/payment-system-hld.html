§ 00

Scope and Requirements

What we are building, and what we deliberately leave out

We are designing a Stripe like payment processor. Merchants integrate with our APIs, customers pay using credit or debit cards, and merchants receive reliable status updates. The difficult part is not creating a row in a database; it is preserving correctness while the external payment network is asynchronous, slow, or uncertain.

Functional
Core requirements
  • Merchant creates a PaymentIntent
  • Customer pays using a card token
  • Merchant reads current payment status
  • Merchant receives status webhooks
Non functional
System guarantees
  • Highly secure card handling
  • Durable and auditable records
  • No double charging on retries
  • Scales beyond 10,000 TPS
⚠️
Out of scope: saved payment methods, refunds, disputes, recurring subscriptions, merchant payouts, and full ledger accounting. The architecture leaves room to add them later.
§ 01

Core Mental Model

Separate merchant intent from money movement attempts

The cleanest way to reason about a payment system is to separate the business level payment from the individual attempts to move money.

Merchant Orderorder_123
→
PaymentIntentpay_intent_789
↓ one to many
Transaction 1declined
Transaction 2processing
Transaction 3succeeded
PaymentIntent
The merchant's intention to collect a fixed amount and currency. It is the simplified object shown to the merchant.
Transaction
One charge attempt associated with the intent. A PaymentIntent can have multiple transactions.
TransactionAttempt
One network request/retry for the same transaction. It preserves attempt level audit detail without creating a new charge identity.
✅
Key distinction: a retry after a timeout is usually another attempt for the same Transaction, not a brand new Transaction.

§ 05

Merchant Authentication

Machine to machine credentials, not a human login flow

A merchant is a client of our platform, but its backend makes automated server to server calls. The simplest model is a merchant API key. An OAuth client credentials flow is also valid when short lived access tokens are preferred.

1
Merchant registers and receives merchant_id + API secret
│
2
Merchant calls API with Authorization: Bearer sk_live_...
│
3
API Gateway validates key, status, expiry and scopes
│
4
Gateway derives merchant_id and injects trusted identity internally
│
5
Gateway applies per merchant rate limits and routes the request
Baseline
API key
  • Simple machine to machine integration
  • Key prefix identifies record
  • Secret is securely hashed/encrypted
  • Support revoke, rotate and expire
Stronger option
HMAC request signing
  • Sign method + path + body hash + timestamp
  • Detect body tampering
  • Timestamp/nonce reduces replay risk
  • Useful for high security integrations
🔐
Never trust a merchant_id from the request body. Derive merchant identity from the authenticated credential.

The first secure call must carry the raw card details somewhere; otherwise tokenization cannot happen. The design goal is not “the card never crosses a network.” The goal is that the card travels only through a small, hardened PCI compliant path.

§ 09

Transaction Safety & Double Charge Prevention

Idempotency at every retry boundary

The dangerous scenario is simple: we send a $50 charge, the bank processes it, but the response packet is lost. If we retry as a fresh charge, the customer may pay twice.

Wrong
Blind retry
  • Create a new transaction after timeout
  • Generate a new external request ID
  • Network may create a second charge
  • Customer sees duplicate debit
Correct
Stable identity
  • Keep the same transaction_id
  • Reuse the same external idempotency key
  • Record a new transaction_attempt
  • Return/recover the original result
External request identity
external_idempotency_key = "txn_82f1..."

// Retry 1
charge(amount=2499, key="txn_82f1...")

// Retry 2 after timeout — SAME key
charge(amount=2499, key="txn_82f1...")

Safe processing also requires conditional state transitions. Use a version column or compare and set update so two workers cannot independently finalize the same transaction.

Optimistic state transition
UPDATE transaction
SET status = 'SUCCEEDED', version = version + 1
WHERE transaction_id = ?
  AND status IN ('PENDING', 'SENT', 'UNKNOWN')
  AND version = ?;
✅
Safety rule: the same merchant request returns the same internal result, and the same external charge identity is reused across retries.
§ 10

Timeouts, UNKNOWN State & Reconciliation

Accept uncertainty instead of inventing certainty

REQUEST SENT
TIMEOUT
UNKNOWN
RECONCILIATION
FINAL RESULT

A timeout means only that our system did not receive a definitive answer. It does not mean the bank failed to charge. Therefore, timeout transitions to UNKNOWN or PENDING_RECONCILIATION, never directly to FAILED.

1
Retry status query / charge with the same idempotency key
│
2
Listen for network callbacks or asynchronous settlement events
│
3
Periodic reconciliation compares our UNKNOWN records with network records
│
4
Match using external reference / idempotency key / amount / merchant
│
5
Atomically finalize SUCCEEDED or FAILED and emit an audit event
⚠️
Do not create a new charge transaction merely because “enough time passed.” Create a new transaction only after the previous result is definitely failed/canceled, or after reconciliation confirms no charge occurred.
§ 11

Durability & Complete Auditability

Current state, immutable history, and reliable event propagation

Payment correctness requires more than the latest status. Months later, we may need to prove exactly when the charge was created, sent, timed out, retried, authorized, settled and reported to the merchant.

Mutable view
Current state tables
  • Fast reads for merchant APIs
  • Current PaymentIntent status
  • Current Transaction status
  • Optimized for operations
Immutable view
Event & attempt history
  • Append only payment_event_log
  • Append only transaction_attempt
  • Reconstruct full lifecycle
  • Supports disputes and compliance

For every important state change, write the following in one database transaction:

Atomic commit
BEGIN;

UPDATE transaction
SET status = 'SUCCEEDED', version = version + 1
WHERE transaction_id = 'txn_123';

INSERT INTO payment_event_log
(event_id, aggregate_id, event_type, old_status, new_status, created_at)
VALUES
('evt_9', 'txn_123', 'TRANSACTION_SUCCEEDED',
 'SENT', 'SUCCEEDED', now());

INSERT INTO outbox
(event_id, partition_key, event_type, payload, status)
VALUES
('evt_9', 'pi_456', 'TRANSACTION_SUCCEEDED', ..., 'NEW');

COMMIT;

CDC or an outbox publisher emits committed rows to Kafka. If Kafka is unavailable, the database commit still survives and the event can be published later. Consumers deduplicate by event_id.

WAL / durable commitMulti AZ replicationAppend only audit logTransactional outboxReplayable KafkaBackups + PITR
💡
Responsibility split: current tables answer “what is the state now'”, the audit log answers “how did we get here'”, and outbox/Kafka answers “how do committed changes reliably reach other services'”

Kafka Partitioning & Ordering

Preserve order per payment while scaling different payments in parallel

Partition key
partition_key = payment_intent_id

All events belonging to one PaymentIntent land in the same partition:

Ordered event stream
PAYMENT_INTENT_CREATED
→ TRANSACTION_CREATED
→ TRANSACTION_SENT
→ TRANSACTION_UNKNOWN
→ TRANSACTION_SUCCEEDED
→ WEBHOOK_REQUESTED

Different PaymentIntents are spread across partitions and processed concurrently. Within a consumer group, a partition is owned by one consumer at a time; ownership can move after failure or rebalance. The guarantee is same payment → same partition → ordered processing, not “same permanent server.”

Scaling Beyond 10,000 TPS

Scale each bottleneck independently

10K+
Payment requests / sec
~5 MB/s
Raw row data at 500 B each
~432 GB/day
Raw data before indexes/replicas
3×
Kafka replication
Stateless services
Run many API Gateway, PaymentIntent, Transaction, Webhook and Reconciliation instances behind load balancers. Autoscale on CPU, latency, queue lag and error rate.
Kafka
Use enough partitions for peak event volume and headroom, benchmarked in production. Partition by payment_intent_id and scale consumers through consumer groups.
Database writes
Start with a strongly consistent relational primary. As write volume grows, shard by merchant_id so PaymentIntent, Transaction, attempts and events for a merchant remain colocated.
Database reads
Use read replicas for history/reporting. Read recent status from primary or use read your writes routing when replica lag could expose stale financial state.
Caching
Cache merchant configuration, API key metadata and rate limit state. Do not make Redis the source of truth for mutable financial status.
Archival
Partition operational tables by time, move older immutable events/transactions to object storage, and keep an indexed retrieval path for audits and disputes.
Backpressure
Bound worker concurrency per external network, use circuit breakers and queues, and degrade gracefully when a bank is slow instead of exhausting all threads.
💡
Sharding key choice: merchant_id is a strong operational shard key because most ownership, authorization and reporting queries are merchant scoped. Kafka still uses payment_intent_id for per payment ordering.

Final Interview Summary

A merchant authenticates using an API key and creates a PaymentIntent. The customer enters card details through our secure hosted iframe or JavaScript SDK. Raw card data goes directly to a restricted PCI compliant tokenization service over TLS, and the rest of the platform uses only a card token.

When the merchant confirms payment, the Transaction Service creates a Transaction and a TransactionAttempt. Payment state, an immutable audit event, and an outbox event are committed atomically. CDC/outbox then publishes the committed event to Kafka, partitioned by payment_intent_id so each payment's state transitions remain ordered.

Transaction workers call the external network using a stable idempotency key derived from the transaction identity. Retries reuse the same key. A timeout is never treated as failure; it becomes UNKNOWN and is later resolved by callbacks, status queries or reconciliation against the network's final records.

The database remains the source of truth. Current state tables provide fast reads, append only event and attempt tables provide complete auditability, the outbox guarantees reliable propagation, and Kafka enables independently scalable consumers such as webhooks, reconciliation, fraud and analytics.

To scale past 10,000 TPS, services stay stateless behind load balancers, Kafka uses many partitions and consumer groups, the relational database is optimized and eventually sharded by merchant_id, read replicas serve history/reporting, Redis caches non financial metadata, and old immutable records are archived to object storage.
