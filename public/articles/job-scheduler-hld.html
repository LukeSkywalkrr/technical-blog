S 00

Core Design Choice

DB is durable, queue is precise, worker claim is the safety gate

The important split is between job definition and job execution. A job definition says what should run. A job execution is one concrete scheduled run of that job.

The durable database remains the source of truth. The queue is a near time delivery mechanism that wakes workers close to the scheduled time. A worker must still claim the execution in the database before running it.

KEY
Interview line: execution table by time, queue for precision, worker lease for safety, idempotency for duplicates.
Time bucket partitions Delayed queue Atomic lease Idempotent task

S 03

Time Bucket Calculation

Use epoch time directly, not calendar subtraction

The scheduler query is time based. It asks for executions due between now and a short lookahead window. That means the execution table should be organized around time.

A scheduler with a 10 minute lookahead scans the active buckets, then scans only the shard ids it owns inside each bucket.
Bucket calculation 
bucket_size_ms = 5 * 60 * 1000
time_bucket = floor(scheduled_at_epoch_ms / bucket_size_ms) * bucket_size_ms

Epoch time already includes date, month, year, and time zone normalization. Today at 10 AM and tomorrow at 10 AM land in different buckets because their epoch timestamps are different.

S 04

Partition Strategy

Do not shard the scheduler path by job id

Job can be partitioned by job_id because job lookup is identity based. JobExecution should be partitioned by time_bucket + shard_id because scheduler lookup is due time based.

Good for job definition
PK = job_id
  • Fetch job details
  • Pause or delete job
  • Update schedule expression
Good for scheduler scan
PK = time_bucket#shard_id
  • Find executions due soon
  • Avoid hot partitions
  • Assign shard ranges to schedulers
DynamoDB or Cassandra style key 
PK = time_bucket#shard_id
SK = scheduled_at#execution_id

Example:
PK = 1783400400#shard_17
SK = 1783400525#exec_abc
1
scheduled_at maps to time_bucket
|
2
execution_id maps to shard_id using hash(execution_id) % N
|
3
partition key becomes time_bucket#shard_id
|
4
scheduler scans assigned shard ids inside due buckets
WHY
At 10k jobs per second, one hourly bucket can contain 36 million executions. Splitting the bucket into many shards prevents one partition from absorbing the whole write and scan load.

The direct enqueue path reduces latency for jobs that are due soon. It does not replace the scheduler scanner. The scanner is still the backstop that finds durable executions and pushes them into the queue later.

S 06

Scheduler Service

Scheduler instances own time bucket shards, not job ids

Each scheduler instance is assigned a range of shard ids. Every few seconds, it scans the buckets that intersect the lookahead window, such as now to now plus 5 minutes.

Scheduler 1
shard_0 to shard_9

Scans due executions for its assigned shards and pushes them to the delayed queue.

Scheduler 2
shard_10 to shard_19

Does the same scan for another shard range. Failover can rebalance ranges through service discovery or leases.

Postgres index option 
CREATE INDEX idx_execution_due
ON t_job_execution (time_bucket, shard_id, scheduled_at)
WHERE status = 'SCHEDULED';

For very large scale, a DynamoDB or Cassandra style model with time_bucket#shard_id as the partition key is more natural. For a Postgres answer, mention time based partitioning and sharding options such as Citus.

S 07

Delayed Queue

Use SQS delay or Redis sorted sets for near time execution

The delayed queue is not the long term scheduler. It is the near time precision layer. The durable source of truth remains JobExecution.

SQS delayed message
delay = scheduled_at - now
then
SendMessage DelaySeconds
then
Worker long polls visible messages
then
Worker claims JobExecution
Redis sorted set
score = scheduled_at_epoch_ms
then
ZADD scheduler:bucket:shard
then
Lua pops due execution ids
then
Worker claims JobExecution

When using SQS, the scheduler sets DelaySeconds = scheduled_at - now. The message is not visible to workers until the delay expires.

Concept Meaning
DelaySeconds Hides a new message until the scheduled time
VisibilityTimeout Hides a received message while a worker is processing it
DeleteMessage Acknowledges success after the worker finishes
DLQ Captures poison messages after too many receives
SQS
Workers are not called back by SQS. They continuously long poll and receive messages when those messages become visible.
REDIS
With Redis, workers or queue consumers use a Lua script to atomically find due items with ZRANGEBYSCORE, remove them with ZREM, and then claim the execution in the database.

SQS delay is capped at 15 minutes, so the scheduler should only enqueue near term executions. Redis sorted sets also work best as near time queues because the database can rebuild them if Redis loses data.

S 08

Worker Lease

The database decides who is allowed to run the execution

A worker receiving a queue message is not enough. It must atomically claim the execution row. Only the worker that wins the conditional update is allowed to execute the task.

Atomic claim 
UPDATE t_job_execution
SET status = 'IN_PROGRESS',
    locked_by = :worker_id,
    lock_until = :now_plus_5_minutes,
    version = version + 1,
    attempt_count = attempt_count + 1
WHERE execution_id = :execution_id
  AND status IN ('SCHEDULED', 'QUEUED', 'RETRYING')
  AND (lock_until IS NULL OR lock_until < :now)
RETURNING version;

If the update returns a row, the worker runs. If it returns nothing, another worker already owns the execution, or the execution is no longer runnable.

S 09

Duplicate Enqueue

Duplicates are acceptable when claiming is idempotent

A near term execution can enter the queue from the API, the scheduler scanner, or a retry path. This means duplicate queue messages are normal.

1
Two messages contain the same execution_id
|
2
Worker A claims row and moves it to IN_PROGRESS
|
3
Worker B claim returns no row
|
4
Worker B deletes or ignores the duplicate queue message
OK
Clean answer: duplicate enqueue is acceptable because execution claim is idempotent and conditional.
S 10

Retry and Failure Handling

Retries create a later runnable execution state

When a task fails, the worker updates the execution with failure metadata and computes the next retry time. The retry can reuse the same execution row with status RETRYING, or create a separate attempt row if audit history is required.

Transient failure
Retry later
  • Increment attempt_count
  • Store last_error
  • Set next scheduled_at with backoff
  • Enqueue when inside lookahead
Permanent failure
Move to failed
  • Mark FAILED
  • Stop retrying after max attempts
  • Send queue message to DLQ
  • Expose error in user view

The task itself should receive execution_id as an idempotency key. If a worker crashes after doing external work but before saving completion, a retry should not duplicate the side effect.

S 11

Recurring Jobs

Generate the next execution after the current one completes

For recurring jobs, the system keeps one Job definition and creates one JobExecution row per run. The next execution is calculated from the cron or interval expression after the current run reaches a terminal state.

Recurring completion 
complete(exec_1)
if Job.status == ACTIVE:
    next_time = cron.next_after(exec_1.scheduled_at)
    insert JobExecution(job_id, scheduled_at = next_time)
NOTE
Use a transaction or unique key around job_id + scheduled_at so a duplicate completion path cannot create the same next execution twice.
S 12

Redis Sorted Set Deep Dive

Sorted sets can power the near time queue, while DB remains the source of truth

A Redis sorted set works well as a near time priority queue. The score is scheduled_at_epoch_ms, and the value is execution_id.

Redis key model 
key   = scheduler:{time_bucket}:{shard_id}
score = scheduled_at_epoch_ms
value = execution_id

ZRANGEBYSCORE key -inf now LIMIT 0 N
ZREM key execution_id

Use a Lua script to read and remove due items atomically. Then the worker still claims the execution in the database before running it.

RULE
If Redis loses data, the scheduler can rebuild the near time queue by scanning due executions from the database again.
S 13

User Monitoring

Do not compromise scheduler partitioning for user reads

Scheduler reads and user reads are different access patterns. The scheduler needs executions due soon. A user dashboard needs executions for one user.

Scheduler path
time_bucket#shard_id

Optimized for scanning near future executions across the system.

User path
user_id

Optimized for listing jobs, execution history, status, and error details for one account.

Read model options 
DynamoDB GSI:
PK = user_id
SK = scheduled_at#execution_id

Postgres index:
CREATE INDEX idx_user_execution
ON t_job_execution (user_id, scheduled_at DESC);

At very large scale, this can become a separate read model such as UserJobExecutionView. That keeps the scheduler write and scan path clean.

S 14

Final End to End Flow

All paths unified

Create path

Create 
Client -> Job API
  -> validate schedule and parameters
  -> INSERT Job
  -> compute first scheduled_at
  -> INSERT JobExecution
  -> if inside lookahead, enqueue with delay
  -> return job_id

Scheduler path

Schedule 
Scheduler instance
  -> scans assigned time_bucket + shard_id ranges
  -> finds SCHEDULED executions due within lookahead
  -> marks QUEUED or records enqueue marker
  -> sends message to delayed queue

Worker path

Run 
Worker long polls queue
  -> receives visible execution_id
  -> conditionally claims JobExecution
  -> runs task with execution_id idempotency key
  -> marks COMPLETED or RETRYING or FAILED
  -> deletes queue message after successful handling
  -> creates next execution for recurring jobs

Final Interview Summary

A scalable job scheduler has two main durable entities: Job for the job definition and JobExecution for each actual run.

Job can be partitioned by job_id, but JobExecution should be partitioned by time_bucket + shard_id because the scheduler query is due time based.

The scheduler scans its assigned time bucket shards for executions due in a short lookahead window and pushes them to a delayed queue such as SQS. Workers long poll the queue, then claim execution rows with an atomic conditional update before running.

Duplicate enqueue is acceptable because the database claim is idempotent. If a worker crashes, SQS visibility timeout and the database lease allow another worker to retry. Every task receives execution_id as an idempotency key. For recurring jobs, the next execution is inserted only after the current execution completes.
